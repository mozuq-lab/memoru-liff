# Feature Specification: カード検索

**Feature Branch**: `001-card-search`  
**Created**: 2026-02-28  
**Status**: Draft  
**Input**: User description: "カード検索: カードが増えた際に、キーワードで素早く目的のカードを見つけられる機能"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - キーワード検索でカードを素早く見つける (Priority: P1)

ユーザーがカード一覧画面を開き、検索バーにキーワードを入力すると、カードの表面・裏面のテキストがリアルタイムでフィルタリングされ、該当するカードだけが表示される。50枚以上カードが増えた学習者でも、目的のカードを数秒で見つけられる。

**Why this priority**: カードが増えるとスクロールによる目視確認が限界になる。検索はアプリの基本的な使いやすさに直結し、それだけで独立したMVPとして価値を提供できる。

**Independent Test**: カード一覧画面で検索バーにテキストを入力し、表示されるカードが絞り込まれることを確認。カードを持つユーザーがログインするだけで完全にテスト可能。

**Acceptance Scenarios**:

1. **Given** 複数枚のカードが登録された状態で、**When** 検索バーに任意のキーワードを入力したとき、**Then** カードの表面または裏面にそのキーワードを含むカードのみが表示される
2. **Given** 検索バーにキーワードが入力されている状態で、**When** キーワードを削除して空にしたとき、**Then** 全カードが再び表示される
3. **Given** 検索バーにキーワードを入力している状態で、**When** 一致するカードが存在しない場合、**Then** 「該当するカードがありません」のような空状態メッセージが表示される
4. **Given** 検索バーにキーワードを入力している状態で、**When** カードが表示されたとき、**Then** マッチした文字列部分が視覚的にハイライトされて表示される

---

### User Story 2 - 復習状態でカードをフィルタリングする (Priority: P2)

ユーザーがカード一覧画面でフィルターチップを使って、「今日が期日（due）」「学習中」「新規」といった復習状態でカードを絞り込む。特定の状態のカードだけを把握したい場合に役立つ。

**Why this priority**: キーワード検索が最優先だが、復習状態によるフィルタリングは学習管理に有益。既存の due/all タブ切り替えを発展させた形で独立して実装・テスト可能。

**Independent Test**: フィルターチップ（due / 学習中 / 新規）を選択し、対応する状態のカードのみが表示されることを確認。

**Acceptance Scenarios**:

1. **Given** カード一覧画面が表示された状態で、**When** 「due（期日）」フィルターを選択したとき、**Then** 今日以前に次回復習日が設定されているカードのみが表示される
2. **Given** フィルターが選択されている状態で、**When** キーワード検索と組み合わせたとき、**Then** フィルター条件とキーワードの両方を満たすカードが表示される
3. **Given** 複数のフィルターが選択可能な場合で、**When** フィルターを切り替えたとき、**Then** 即座に表示結果が更新される

---

### User Story 3 - ソート順を切り替えてカードを整理する (Priority: P3)

ユーザーがカード一覧の表示順を「作成日」「次回復習日」「習熟度（ease factor）」などで並び替え、学習状況の把握と優先度判断を行う。

**Why this priority**: 並び替えは既存の絞り込み機能を補完するが、学習の根本体験には直結しない。検索・フィルタリングが完成した後で追加できる。

**Independent Test**: ソートドロップダウンで並び替え条件を選択し、カードの表示順が変わることを確認。

**Acceptance Scenarios**:

1. **Given** カード一覧が表示されている状態で、**When** 「次回復習日」ソートを選択したとき、**Then** 次回復習日が近い順にカードが並び替えられる
2. **Given** 「習熟度（低い順）」ソートを選択した場合で、**When** カードが表示されたとき、**Then** ease factor が低いカード（苦手なカード）が上位に表示される
3. **Given** ソート順とキーワード検索を組み合わせた状態で、**When** ソートを変更したとき、**Then** 検索結果の順序が変更されるが、フィルタリング結果は維持される

---

### Edge Cases

- カードが0枚の状態で検索バーを操作したとき、空のリストが正常に表示される
- 検索キーワードに特殊文字（`<`, `>`, `&`, `"` など）が含まれる場合、XSSを引き起こさず安全に処理される
- 非常に長いキーワード（100文字以上）を入力しても、UIが崩れずに動作する
- 大文字・小文字の違いは区別しない（例：「ABC」と「abc」で同じ結果を返す）
- 全角・半角の区別は区別しない（例：「ABC」と「ＡＢＣ」で同じ結果を返す）
- カードが多数（500枚以上）存在する場合でも、フィルタリング結果が即座（300ms以内）に表示される

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: カード一覧画面の上部に常設の検索バーを表示しなければならない
- **FR-002**: ユーザーが検索バーに入力するたびに、カードの表面（front）と裏面（back）テキストをリアルタイムでフィルタリングしなければならない
- **FR-003**: 検索は大文字・小文字を区別してはならない（case-insensitive）
- **FR-004**: 検索は全角・半角を区別してはならない（日本語・英語混在対応）
- **FR-005**: 検索にマッチしたテキスト部分をハイライト表示しなければならない
- **FR-006**: 検索バーが空のとき、すべてのカードを表示しなければならない
- **FR-007**: 検索結果が0件の場合、空状態メッセージを表示しなければならない
- **FR-008**: 復習状態（due / 学習中 / 新規）によるフィルターチップを提供しなければならない
- **FR-009**: キーワード検索とフィルタリング条件は組み合わせて適用されなければならない
- **FR-010**: ソート順（作成日 / 次回復習日 / 習熟度）の切り替えを提供しなければならない
- **FR-011**: Phase 1 はフロントエンドのみのクライアントサイドフィルタリングとして実装する。既にロード済みのカードデータに対して絞り込みを行う
- **FR-012**: 検索バーにはクリアボタンを設け、ワンタップで検索をリセットできなければならない

### Key Entities

- **カード (Card)**: 表面テキスト（front）・裏面テキスト（back）・復習状態（due_date, ease_factor, status）を持つ学習単位。検索・フィルタリングの対象
- **フィルター状態 (Filter State)**: ユーザーが選択したキーワード・復習状態・ソート条件の組み合わせ。URLパラメータに反映し、ページ遷移後も維持される（任意）

### Assumptions

- Phase 1 ではクライアントサイドフィルタリングのみ実装し、バックエンドAPIの変更は行わない
- 既存の `GET /cards` API をそのまま使用し、全カードを取得後にフロントエンドで絞り込む
- デッキによるフィルタリングは「カテゴリ（デッキ）管理」機能（別フィーチャー）実装後に追加可能な拡張ポイントとして設計する
- モバイルファーストのUI（LINE LIFF環境での利用を前提）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが検索バーへの入力開始から結果表示まで300ms以内に完了する
- **SC-002**: 500枚のカードを持つユーザーが目的のカードをキーワード1〜2語で3クリック以内に見つけられる
- **SC-003**: 検索・フィルター・ソートの操作において、画面の読み込みや再取得が発生せず即座にUIが反応する
- **SC-004**: 検索入力中のハイライト表示により、ユーザーがマッチ部分を一目で識別できる
- **SC-005**: XSSを含むセキュリティ上の問題が検索機能から発生しない（手動テスト + 静的解析で確認）

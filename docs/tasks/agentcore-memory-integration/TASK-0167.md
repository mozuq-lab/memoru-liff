## タスク概要

`TutorService.send_message()` をリファクタリングし、会話履歴の管理を SessionManager に委譲する。DynamoDB からの全履歴取得 → AI に渡す方式から、SessionManager が自動的に履歴を復元・保存する方式に移行する。

## 実装詳細

### 1. send_message のリファクタリング 🔵

**信頼性**: 🔵 *設計文書 dataflow.md メッセージ送信フローより*

**実装ファイル**: `backend/src/services/tutor_service.py`

変更前:
```python
# DynamoDB から messages 全件取得
messages = item.get("messages", [])
conversation = [{"role": m["role"], "content": m["content"]} for m in messages]
conversation.append({"role": "user", "content": content})
ai_content, related_cards = self.ai_service.generate_response(system_prompt, conversation)
```

変更後:
```python
# SessionManager が履歴を自動復元・保存
sm = self._create_session_manager(session_id, user_id)
with sm:
    ai_content, related_cards = self.ai_service.generate_response(
        system_prompt, content, session_manager=sm
    )
```

### 2. メタデータのみ DynamoDB 更新 🔵

**信頼性**: 🔵 *REQ-007・設計文書 architecture.md より*

会話履歴は SessionManager が管理するため、DynamoDB では `message_count`, `updated_at` のみ更新。
`messages` フィールドへの追記は不要（DynamoDB バックエンド時は SessionManager が内部で処理）。

### 3. 既存チェックロジックの維持 🔵

**信頼性**: 🔵 *REQ-202・既存実装より*

- セッション状態チェック（ended/timed_out → SessionEndedError）
- タイムアウトチェック（30分）
- メッセージ上限チェック（20往復）

これらは既存ロジックをそのまま維持。

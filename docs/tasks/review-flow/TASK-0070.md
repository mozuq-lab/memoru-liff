## 実装詳細

### 1. ローカル状態管理 🔵

**信頼性**: 🔵 *architecture.md のローカル状態設計・dataflow.md の状態遷移図より*

ReviewPage のローカル状態を useState で管理する。

```typescript
const [cards, setCards] = useState<DueCard[]>([]);
const [currentIndex, setCurrentIndex] = useState(0);
const [isFlipped, setIsFlipped] = useState(false);
const [isSubmitting, setIsSubmitting] = useState(false);
const [reviewedCount, setReviewedCount] = useState(0);
const [isComplete, setIsComplete] = useState(false);
const [isLoading, setIsLoading] = useState(true);
const [error, setError] = useState<string | null>(null);
```

**実装ファイル**: `frontend/src/pages/ReviewPage.tsx`

### 2. 復習カード取得 🔵

**信頼性**: 🔵 *要件定義REQ-RV-040・既存API仕様・dataflow.md の復習開始シーケンスより*

- useEffect で初回マウント時に `cardsApi.getDueCards()` を呼び出し
- 成功: cards に設定、isLoading=false
- 失敗: error に設定、isLoading=false
- 0枚: 空状態メッセージ表示

**実装ファイル**: `frontend/src/pages/ReviewPage.tsx`

### 3. 採点送信処理 🔵

**信頼性**: 🔵 *要件定義REQ-RV-041・dataflow.md の採点送信シーケンスより*

- handleGrade(grade):
  1. isSubmitting=true
  2. `reviewsApi.submitReview(currentCard.card_id, grade)` 呼び出し
  3. 成功: reviewedCount++, moveToNext()
  4. 失敗: error 表示（再送信可能）
  5. isSubmitting=false

### 4. スキップ処理 🔵

**信頼性**: 🔵 *要件定義REQ-RV-020〜021・dataflow.md のスキップフローより*

- handleSkip():
  1. API送信なし
  2. reviewedCount は増やさない
  3. moveToNext()

### 5. カード進行処理 🔵

**信頼性**: 🔵 *dataflow.md の状態遷移図より*

- moveToNext():
  1. isFlipped=false（次のカードは表面から）
  2. currentIndex が最後なら isComplete=true
  3. そうでなければ currentIndex++

### 6. ナビゲーションバー非表示 🔵

**信頼性**: 🔵 *要件定義REQ-RV-401・ユーザヒアリングより*

- ReviewPage ではナビゲーションバーを非表示にする
- 実装方法: 条件付きレンダリング（App.tsx で pathname チェック）または ReviewPage 独自のレイアウト

### 7. 「戻る」ボタン 🔵

**信頼性**: 🔵 *要件定義REQ-RV-402・ユーザヒアリングより*

- ページ上部に「戻る」ボタン（← アイコン）
- `useNavigate()` の `navigate(-1)` で前画面に戻る
- 採点済みカードは API 送信済みなので保存されている

### 8. エラーハンドリング 🟡

**信頼性**: 🟡 *REQ-RV-103〜104・既存実装パターンから妥当な推測*

- 初期読み込みエラー: エラーメッセージ + リトライボタン
- 採点送信エラー: エラーメッセージ表示（同じボタンで再送信可能）

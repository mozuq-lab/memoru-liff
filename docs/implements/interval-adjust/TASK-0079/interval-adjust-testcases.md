  - **堅牢性の確認**: 状態遷移を繰り返してもUI表示が正確であること
- 🟡 既存テストパターン（テストケース3）と要件定義 REQ-201 から妥当な推測

### TC-F17: 連続でプリセットボタンを押した場合の挙動

- **テスト名**: プリセットボタン連続タップで最後のAPI呼び出しのみが処理される
  - **境界値の意味**: 高速連続操作時のUI制御の一貫性
  - **境界値での動作保証**: 二重送信防止により最初のAPI完了前に2回目の操作がブロックされること
- **入力値**: 「1日」ボタンタップ後、即座に「30日」ボタンタップを試みる（API未完了状態）
  - **境界値選択の根拠**: ボタン無効化によるガードの検証
  - **実際の使用場面**: ユーザーが操作を迷って素早く押し直す場面
- **期待される結果**: `mockUpdateCard` が1回だけ呼ばれる（2回目のクリックは disabled で無視される）
  - **境界での正確性**: disabled 状態でのクリックイベントが発火しないこと
  - **一貫した動作**: API完了後にボタン有効化 → 再度操作可能
- **テストの目的**: 連続操作耐性の確認
  - **堅牢性の確認**: 高速操作でもデータ不整合が発生しないこと
- 🟡 要件定義 REQ-202・EDGE-002・データフロー図から妥当な推測

describe('復習間隔プリセットボタン', () => {

  // 【テスト内容】: ボタンクリック後にmockUpdateCardの呼び出し引数を検証
  // 【期待される動作】: updateCard(cardId, { interval: 7 }) が呼ばれる
  // 🔵 要件定義 REQ-002 より
  it('プリセットボタン「7日」タップでAPIがinterval=7で呼ばれる', async () => {
    const user = userEvent.setup();
    const updatedCard = { ...mockCard, interval: 7 };
    mockUpdateCard.mockResolvedValue(updatedCard);

    // 【テストデータ準備】: カード詳細ページをレンダリング
    renderCardDetailPage();

    await waitFor(() => {
      expect(screen.getByTestId('card-detail')).toBeInTheDocument();

    await user.click(screen.getByTestId('preset-button-7'));

    // 【結果検証】: updateCard APIが正しい引数で呼ばれること
    // 【検証項目】: 第1引数がcard_id、第2引数に interval: 7 が含まれる 🔵
    await waitFor(() => {
      expect(mockUpdateCard).toHaveBeenCalledWith('card-1', { interval: 7 });
    });

    });

# Memoru LIFF 機能バックログ

作成日: 2026-02-28

## 概要

今後実装を検討する機能の一覧。優先度と実装難易度を整理し、開発計画の参考とする。

---

## 1. カテゴリ（デッキ）管理

### 目的
カードをデッキ（カテゴリ）単位でグループ化し、整理・復習を効率化する。

### 現状
- cards テーブルに `deck_id` フィールドが既に存在するが、未使用
- AI カード生成時に `suggested_tags` が返されるが、デッキとの連携はない

### 機能要件
- デッキの CRUD（作成・一覧・更新・削除）
  - デッキ名、説明、アイコンカラー（任意）を設定可能
  - デッキ削除時はカードを「未分類」に移動（カードは削除しない）
- カードをデッキに割り当て / 移動
  - カード作成時にデッキを選択可能
  - 既存カードのデッキ変更（カード編集画面 or 一括操作）
- デッキ単位の操作
  - デッキを指定して復習セッションを開始
  - デッキごとの復習進捗（due カード数）を表示
- 「未分類」デッキ
  - デッキ未割り当てのカードを自動的にグループ化

### UI イメージ
- ホーム画面にデッキ一覧セクションを追加（カード枚数・due 数を表示）
- デッキタップでデッキ内カード一覧へ遷移
- カード一覧画面にデッキフィルタードロップダウンを追加

### データモデル変更
- **decks テーブル（新規）**
  - PK: `user_id`, SK: `deck_id`
  - フィールド: `name`, `description`, `color`, `card_count`, `created_at`, `updated_at`
- **cards テーブル**
  - 既存の `deck_id` フィールドを活用（デフォルト: `"uncategorized"`）

### API エンドポイント（案）
- `GET /decks` — デッキ一覧取得
- `POST /decks` — デッキ作成
- `PUT /decks/{deckId}` — デッキ更新
- `DELETE /decks/{deckId}` — デッキ削除
- `PUT /cards/{cardId}` — 既存エンドポイントで `deck_id` 更新に対応
- `GET /cards?deck_id=xxx` — デッキでフィルタリング
- `GET /cards/due?deck_id=xxx` — デッキ指定で due カード取得

### 実装難易度
中程度 — `deck_id` が既にスキーマにあるため、バックエンドの拡張は比較的容易。フロントエンドの UI 追加がメイン作業。

## 2. カード検索

### 目的
カードが増えた際に、キーワードで素早く目的のカードを見つけられるようにする。

### 現状
- カード一覧画面（CardsPage）に due/all のタブ切り替えはあるが、検索機能はない
- バックエンド `GET /cards` はページネーション対応済み

### 機能要件
- フロントエンド側のリアルタイム検索（インクリメンタルサーチ）
  - カードの表面（front）・裏面（back）のテキストを対象
  - 入力に応じて即座にフィルタリング
- 検索結果のハイライト表示
  - マッチしたキーワード部分を強調表示
- フィルタリング条件の組み合わせ
  - デッキ（カテゴリ管理と連携）
  - 復習状態（due / 学習中 / 新規）
  - ソート順（作成日 / 次回復習日 / ease_factor）

### UI イメージ
- カード一覧画面の上部に検索バーを常設
- 検索バー下にフィルターチップ（デッキ、状態）を表示
- 検索結果は既存の CardList コンポーネントを再利用

### 実装方針
- **Phase 1**: フロントエンドのみで実装（取得済みカードのクライアントサイドフィルタリング）
- **Phase 2**（カード数が多い場合）: バックエンドに検索 API を追加（DynamoDB Scan + FilterExpression or OpenSearch 連携）

### 実装難易度
低〜中 — Phase 1 はフロントエンドのみで完結。Phase 2 は DynamoDB の制約を考慮する必要あり。

## 3. 学習統計ダッシュボード

### 目的
学習の進捗と傾向を可視化し、モチベーション維持と学習計画の改善を支援する。

### 現状
- `GET /reviews/stats` エンドポイントが存在（基本的な統計を返す）
- reviews テーブルに履歴データが蓄積されている（TTL あり）
- ホーム画面には due カード数のみ表示

### 機能要件

#### 基本統計
- 総カード数 / 学習済みカード数 / 未学習カード数
- 平均 ease_factor（全体の習熟度指標）
- 連続学習日数（ストリーク）

#### 時系列グラフ
- 日別復習カード数（棒グラフ、直近 30 日）
- 日別正答率の推移（折れ線グラフ、grade 3 以上を正答とみなす）
- 週別の復習量推移

#### 分析
- デッキ別の習熟度比較（カテゴリ管理と連携）
- 苦手カード TOP 10（ease_factor が低いカード）
- 今後の復習予測（向こう 7 日間の due カード数予測）

### UI イメージ
- ホーム画面のリデザイン、または専用の「統計」タブを追加
- グラフ描画には軽量ライブラリ（Recharts or Chart.js）を使用
- ストリークは炎アイコン + 日数で目立つように表示

### データモデル変更
- **user_stats テーブル（新規、任意）**
  - PK: `user_id`, SK: `date` (YYYY-MM-DD)
  - フィールド: `reviews_count`, `correct_count`, `total_grade`, `cards_added`
  - 日次集計データを保存（reviews テーブルの TTL 後もデータを保持するため）
- reviews テーブルの TTL 設定を見直し（統計用途で長期保持が必要な場合）

### API エンドポイント（案）
- `GET /stats/summary` — 基本統計（総カード数、ストリーク、平均 ease_factor）
- `GET /stats/daily?from=YYYY-MM-DD&to=YYYY-MM-DD` — 日別復習データ
- `GET /stats/weak-cards?limit=10` — 苦手カード一覧
- `GET /stats/forecast?days=7` — 復習予測

### 実装難易度
中〜高 — 日次集計の仕組み（Lambda スケジュール or レビュー時にインクリメント）と、フロントエンドのグラフ描画が主な作業。

## 4. カード画像対応

### 目的
テキストだけでなく画像を使った暗記カードを作成可能にし、図解・写真による記憶定着を促進する。

### ユースケース
- 解剖学・生物学の図解
- 地図・地理の学習
- 化学構造式
- 外国語の文字（漢字・ハングルなど）の書き順
- 数学の公式やグラフ

### 機能要件
- カードの表面・裏面にそれぞれ画像を 1 枚添付可能（任意）
  - テキスト + 画像の組み合わせ、または画像のみ
- 対応形式: JPEG, PNG, WebP
- ファイルサイズ上限: 5MB / 画像
- 画像のリサイズ・圧縮（サーバーサイド or クライアントサイド）
- 画像の削除・差し替え

### UI イメージ
- カード作成/編集フォームに画像アップロードエリアを追加
- ドラッグ&ドロップ or タップでファイル選択
- プレビュー表示（サムネイル）
- 復習画面で画像をタップすると拡大表示

### アーキテクチャ
- **画像ストレージ**: S3 バケット（`memoru-card-images-{env}`）
  - パス: `{user_id}/{card_id}/{front|back}.{ext}`
- **アップロードフロー**:
  1. フロントエンドが Pre-signed URL を API に要求
  2. API が S3 Pre-signed URL を生成して返却
  3. フロントエンドが Pre-signed URL に直接アップロード
  4. カード保存時に画像 URL を cards テーブルに記録
- **配信**: CloudFront 経由で画像を配信（キャッシュ有効）

### データモデル変更
- **cards テーブル**
  - `front_image_key`: S3 オブジェクトキー（任意）
  - `back_image_key`: S3 オブジェクトキー（任意）

### API エンドポイント（案）
- `POST /cards/{cardId}/upload-url` — Pre-signed URL 取得
  - リクエスト: `{ "side": "front" | "back", "content_type": "image/jpeg" }`
  - レスポンス: `{ "upload_url": "...", "image_key": "..." }`
- `DELETE /cards/{cardId}/image` — 画像削除
  - リクエスト: `{ "side": "front" | "back" }`

### 実装難易度
中〜高 — S3 + CloudFront のインフラ構築、Pre-signed URL の実装、フロントエンドの画像 UI が必要。Lambda のペイロード制限（6MB）を避けるため、直接 S3 アップロード方式が必須。

## 5. AI 要約カード自動生成（URL / PDF 入力）

### 目的
Web ページや PDF ドキュメントの内容を AI が要約し、学習カードを自動生成する。手動でテキストをコピペする手間を省く。

### 現状
- テキスト入力からの AI カード生成は実装済み（`POST /cards/generate`）
- Strands Agents SDK / Bedrock Claude 統合済み

### 機能要件

#### URL 入力
- Web ページの URL を入力すると、ページ内容を取得・要約してカード生成
- 対応: 一般的な Web ページ（HTML）
- JavaScript レンダリングが必要なページは非対応（初期版）
- robots.txt を尊重

#### PDF 入力
- PDF ファイルをアップロードすると、テキストを抽出してカード生成
- ページ範囲を指定可能（例: 1-10 ページ）
- ファイルサイズ上限: 10MB
- 対応: テキストベース PDF（スキャン画像 PDF は初期版では非対応）

#### 共通
- 生成前に要約プレビューを表示
- カード候補を表示し、ユーザーが選択して保存（既存フローと同じ）
- 生成カード数を指定可能（5〜20 枚）
- デッキ（カテゴリ）を指定して保存

### UI イメージ
- 既存の GeneratePage を拡張
  - 入力方式のタブ切り替え: 「テキスト」「URL」「PDF」
  - URL タブ: URL 入力欄 + 「取得」ボタン → 要約プレビュー → カード生成
  - PDF タブ: ファイルアップロード + ページ範囲指定 → 要約プレビュー → カード生成
- 要約プレビュー画面（新規）
  - 抽出されたテキストの要約を表示
  - ユーザーが確認後「カード生成」ボタンで進む

### アーキテクチャ

#### URL 処理フロー
1. フロントエンドが URL を API に送信
2. Lambda が URL のコンテンツを取得（`requests` + `BeautifulSoup`）
3. HTML → テキスト変換、不要な要素（ナビ、フッター等）を除去
4. テキストを AI に渡してカード生成（既存のカード生成パイプラインを再利用）

#### PDF 処理フロー
1. フロントエンドが PDF を S3 にアップロード（Pre-signed URL 方式、画像対応と共通化）
2. Lambda が S3 から PDF を取得、テキスト抽出（`pypdf` ライブラリ）
3. 抽出テキストを AI に渡してカード生成

### API エンドポイント（案）
- `POST /cards/generate-from-url`
  - リクエスト: `{ "url": "https://...", "card_count": 10, "deck_id": "xxx" }`
  - レスポンス: 既存の generate レスポンスと同形式
- `POST /cards/generate-from-pdf`
  - リクエスト: `{ "s3_key": "...", "page_range": "1-10", "card_count": 10, "deck_id": "xxx" }`
  - レスポンス: 既存の generate レスポンスと同形式

### 考慮事項
- Lambda のタイムアウト: URL 取得 + AI 生成で 30 秒を超える可能性
  - 対策: Step Functions で非同期処理、またはタイムアウトを延長（最大 15 分）
- コンテンツの長さ制限: AI モデルのコンテキストウィンドウに収まるよう、テキストをチャンク分割 or 要約してから生成
- コスト: 長文の AI 処理はトークン消費が大きいため、利用回数制限を検討

### 実装難易度
高 — URL スクレイピング、PDF テキスト抽出、長文処理の最適化、非同期処理の検討が必要。ただし既存の AI カード生成パイプラインを再利用できる部分は多い。

## 優先度・実装順序の提案

| # | 機能 | 難易度 | 依存関係 | 推奨順序 |
|---|------|--------|----------|----------|
| 1 | カード検索 | 低〜中 | なし | 1st |
| 2 | カテゴリ（デッキ）管理 | 中 | なし | 2nd |
| 3 | 学習統計ダッシュボード | 中〜高 | なし（デッキ連携は後から追加可） | 3rd |
| 4 | カード画像対応 | 中〜高 | S3 インフラ | 4th |
| 5 | AI URL/PDF カード生成 | 高 | 画像対応の S3 基盤を共用可 | 5th |

### 推奨理由
1. **カード検索**は実装コストが低く、既存ユーザーの利便性を即座に向上できる
2. **カテゴリ管理**は既存の `deck_id` を活用でき、他機能（検索フィルタ、統計のデッキ別分析）の基盤になる
3. **統計ダッシュボード**は既存の reviews データを活用でき、モチベーション向上に直結する
4. **画像対応**は S3 インフラの構築が必要だが、PDF 生成と基盤を共有できる
5. **AI URL/PDF 生成**は最も複雑だが、4 の S3 基盤を再利用でき、差別化機能として価値が高い

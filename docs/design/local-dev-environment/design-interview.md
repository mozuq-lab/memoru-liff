# ローカル開発環境構築 設計ヒアリング記録

**作成日**: 2026-02-23
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

要件定義で確定したスコープに基づき、残タスク（JWT フォールバック・DynamoDB 問題）の技術設計方針を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: 設計の作業規模

**質問日時**: 2026-02-23
**カテゴリ**: 優先順位
**背景**: ローカル開発環境は型定義・DBスキーマ・API仕様の新規設計が不要なため、軽量設計が適切と判断

**回答**: 軽量設計（推奨）

**信頼性への影響**:
- architecture.md と dataflow.md のみ作成
- 既存の型定義・DBスキーマ・API仕様は変更不要であることが確認された

---

### Q2: DynamoDB local SigV4 ハング問題の解決アプローチ

**質問日時**: 2026-02-23
**カテゴリ**: 技術選択
**背景**: `amazon/dynamodb-local:latest` が boto3 の SigV4 署名付きリクエストでハングする問題。Docker イメージバージョン変更か認証無効化かの方針決定が必要

**回答**: Docker イメージ変更

**信頼性への影響**:
- アーキテクチャ設計の「DynamoDB Local 接続問題解決」セクションの方針が確定（🔴 → 🔵）
- docker-compose.yaml の `image:` 行を特定バージョンに変更する設計が確定
- フォールバックとして `-disableTelemetry` / ダミー認証情報 / localstack を記録

---

## ヒアリング結果サマリー

### 確認できた事項
- 軽量設計で十分（型定義・DBスキーマ・API仕様は既存のものを使用）
- DynamoDB 問題は Docker イメージバージョン固定で対応

### 設計方針の決定事項
- architecture.md: システム構成図、JWT フォールバック設計、DynamoDB 解決アプローチ
- dataflow.md: 認証フロー、ルーティングフロー、DynamoDB 接続フロー、エラーハンドリング
- DynamoDB 解決は「バージョン固定 → フォールバック（3案）」の段階的アプローチ

### 残課題
- DynamoDB local の安定バージョン番号の特定（タスク実装時に Docker Hub で確認）

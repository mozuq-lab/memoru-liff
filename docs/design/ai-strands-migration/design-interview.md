# AI Strands Migration 設計ヒアリング記録

**作成日**: 2026-02-23
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存の要件定義・実装コード（`bedrock.py`, `handler.py`, `prompts.py`, `srs.py`）を分析し、Strands Agents SDK 移行における具体的なアーキテクチャ設計の方針を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: AI サービス抽象化パターン

**質問日時**: 2026-02-23
**カテゴリ**: アーキテクチャ
**背景**: 現在の `BedrockService` は boto3 直接呼び出しで実装されている。Strands Agents 移行時に `USE_STRANDS` フィーチャーフラグで新旧実装を切り替えるには、共通のインターフェースが必要。Python での抽象化パターン（Protocol / ABC / 単一クラス）の選択が設計の根幹に関わる。

**質問**: "現在のBedrockServiceはboto3直接呼び出しですが、Strands Agents移行時のAIサービス抽象化レイヤーについて、どのパターンを採用しますか？"

**選択肢**:
- Protocolベース（推奨）
- ABCベース
- StrandsService単独

**回答**: **Protocolベース（推奨）** - Python Protocol（typing.Protocol）でAIServiceインターフェースを定義し、BedrockService/StrandsServiceの両方が実装。USE_STRANDSフラグでファクトリが切替

**信頼性への影響**:
- アーキテクチャの `AIService Protocol` 設計が 🔵 に確定
- ファクトリパターンによるサービス切替が 🔵 に確定
- 既存 `BedrockService` は Protocol 準拠にアダプト（後方互換維持）

---

### Q2: Lambda 関数構成

**質問日時**: 2026-02-23
**カテゴリ**: アーキテクチャ
**背景**: ユーザヒアリングで「機能別タイムアウト（カード生成30秒、採点/アドバイス60秒）」が要求された。現在は単一の `api-main` Lambda（タイムアウト30秒）で全エンドポイントを処理。新 AI 機能の追加に際し、Lambda 関数を分離するか統合するかの設計判断が必要。

**質問**: "新しいAI機能（回答採点、学習アドバイス）のAPIエンドポイント設計について、Lambda関数の構成はどうしますか？"

**選択肢**:
- 既存Lambda統合（推奨）
- 機能別Lambda分離
- API Router分離

**回答**: **既存Lambda統合（推奨）** - 現在のapi-main Lambda関数に新エンドポイント追加。タイムアウトは関数全体で最大値（60秒）に統一

**信頼性への影響**:
- Lambda タイムアウト設定が 30秒 → 60秒 に変更（REQ-SM-401 の実現方法が確定 🔵）
- 新エンドポイントは既存ルーターに追加（実装コスト低）
- `template.yaml` の Globals.Function.Timeout 変更のみで対応

---

### Q3: プロンプト管理

**質問日時**: 2026-02-23
**カテゴリ**: 技術選択
**背景**: 既存の `prompts.py`（96行）はカード生成専用のプロンプトテンプレートを含む。回答採点・学習アドバイスの新 AI 機能にはそれぞれ異なるシステムプロンプトとユーザープロンプトが必要。Strands Agents のシステムプロンプト設定方式との整合性も考慮が必要。

**質問**: "Strands Agentsのプロンプト管理について、既存のprompts.pyのパターンをどう拡張しますか？"

**選択肢**:
- 機能別プロンプトモジュール（推奨）
- 単一ファイル拡張
- Strands Agent システムプロンプト直接

**回答**: **機能別プロンプトモジュール（推奨）** - prompts/ ディレクトリを作り、generate.py, grading.py, advice.py に分離。各モジュールはシステムプロンプトとユーザープロンプトテンプレートを含む

**信頼性への影響**:
- プロンプト管理のディレクトリ構成が 🔵 に確定
- 既存 `prompts.py` は `prompts/generate.py` に移動（リファクタリング）
- 各プロンプトモジュールが Strands Agent の `system_prompt` パラメータに対応

---

### Q4: SRS グレード変換

**質問日時**: 2026-02-23
**カテゴリ**: データモデル
**背景**: 既存の SM-2 アルゴリズム（`srs.py`）は grade 0-5 を入力として受け取る。AI 採点機能では、ユーザーの回答を AI が評価して SRS グレードに変換する必要がある。AI からの出力形式と SM-2 グレードへの変換方式の設計が必要。

**質問**: "回答採点のSRSグレード（0-5）変換について、AI出力からグレードへのマッピングはどうしますか？"

**選択肢**:
- AI直接グレーディング（推奨）
- スコアベース変換

**回答**: **AI直接グレーディング（推奨）** - AIに直接SRSグレード(0-5)を出力させ、プロンプト内でSM-2グレードの定義を明示。JSON形式で{grade, reasoning}を返却

**信頼性への影響**:
- AI 採点のレスポンス形式が `{grade: int, reasoning: str}` に確定 🔵
- プロンプト内に SM-2 グレード定義（0=完全忘却 〜 5=完璧）を埋め込む方式が確定
- 後続の `calculate_sm2()` にそのまま渡せる設計

---

### Q5: 学習アドバイスのデータ参照

**質問日時**: 2026-02-23
**カテゴリ**: データモデル
**背景**: 学習アドバイス機能は DynamoDB の復習履歴（Reviews Table）やカードデータ（Cards Table）を分析する必要がある。Strands Agents の `@tool` デコレータでツールとして実装するか、事前にデータを取得してプロンプトに埋め込むかの設計判断が必要。

**質問**: "学習アドバイス機能のDynamoDBデータ参照について、エージェントツールとして実装しますか？"

**選択肢**:
- 事前クエリ（推奨）
- Strands Tool経由
- ハイブリッド

**回答**: **事前クエリ（推奨）** - アドバイスリクエスト前にPythonでDynamoDBから復習履歴を取得し、集計結果をプロンプトに埋め込む。ツール不要でシンプル

**信頼性への影響**:
- Phase 3（新機能）ではツール統合不要に（Phase 4 で別途実装）
- データ集計ロジックは Python サービス層で実装（AIService とは独立）
- プロンプトに含めるデータ量の制限設計が必要（🟡 詳細は実装時に決定）

---

### Q6: エラーハンドリング

**質問日時**: 2026-02-23
**カテゴリ**: アーキテクチャ
**背景**: 既存の `BedrockServiceError` 階層（Timeout / RateLimit / Internal / Parse）は handler.py で HTTP ステータスコードにマッピングされている。Strands Agents は独自のエラーを返す可能性があり、既存の例外階層との統合方針が必要。

**質問**: "Strands Agentsのエラーハンドリングについて、既存のカスタム例外階層（BedrockServiceError等）をどうしますか？"

**選択肢**:
- 統一例外階層（推奨）
- 個別例外維持

**回答**: **統一例外階層（推奨）** - AIServiceErrorを基底クラスとし、既存Bedrock例外も新Strands例外も共通の階層に統合。handler.pyのエラーマッピングを統一

**信頼性への影響**:
- `AIServiceError` 基底クラスの設計が 🔵 に確定
- 既存 `BedrockServiceError` は `AIServiceError` のサブクラスに変更
- handler.py のエラーハンドリングは `AIServiceError` 系列のみキャッチ

## ヒアリング結果サマリー

### 確認できた事項

- AI サービス抽象化: Python Protocol ベースのインターフェース設計
- Lambda 構成: 既存 api-main に統合、タイムアウト 60 秒に拡張
- プロンプト管理: 機能別モジュール分割（prompts/ ディレクトリ）
- SRS グレード変換: AI 直接グレーディング（0-5 + reasoning の JSON）
- データ参照: 学習アドバイスは事前クエリ方式（ツール不要）
- エラーハンドリング: AIServiceError 統一例外階層

### 設計方針の決定事項

- Protocol ベースの疎結合設計により、新旧実装の切替が容易
- 既存 Lambda 統合により、インフラ変更を最小限に抑制
- 機能別プロンプトモジュールにより、AI 機能の独立した開発が可能
- 統一例外階層により、handler.py のエラーハンドリングが簡潔

### 残課題

- Strands Agents SDK の具体的なバージョンとインポートパス
- Ollama プロバイダーのローカルモデル選定（llama3.2 / phi-3 等）
- プロンプトに含める学習履歴データの最大量・フォーマット
- Strands Agents の具体的なエラー型の調査

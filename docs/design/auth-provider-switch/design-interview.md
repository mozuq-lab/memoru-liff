# 認証プロバイダ切り替え 設計ヒアリング記録

**作成日**: 2026-03-02
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

要件定義で確定した「環境変数/SAMパラメータでの Keycloak / Cognito 切り替え」について、設計上の不明点を解消するためのヒアリングを実施しました。

## 質問と回答

### Q1: 既存実装の詳細分析の要否

**カテゴリ**: アーキテクチャ
**背景**: 要件定義フェーズで認証周りの実装を詳細に調査済みだが、追加の分析が必要か確認。

**回答**: 不要。要件定義と設計文書のみで設計を作成する。

**信頼性への影響**:
- 既存実装の理解は十分であることが確認され、設計文書の信頼性が維持された

---

### Q2: Cognito 使用時の LINE Login 統合方式の扱い

**カテゴリ**: アーキテクチャ
**背景**: 要件定義では「設計フェーズで決定」としていた LINE Login 統合方式について、設計文書でどこまで踏み込むか確認。方式によりフロントエンドの認証フローが変わる可能性があるため。

**回答**: 選択肢を併記する。Cognito 外部 IdP 方式と LIFF トークン交換方式の両方を設計文書に記載し、実装時に決定する。

**信頼性への影響**:
- LINE Login 統合の設計は 🟡（選択肢併記）として記載
- コア設計（OIDC 切り替え部分）の信頼性には影響なし

---

## ヒアリング結果サマリー

### 確認できた事項
- 既存実装の追加分析は不要（要件定義フェーズで十分に調査済み）
- LINE Login 統合は選択肢併記で実装時に決定

### 設計方針の決定事項
- `backend/template.yaml`: パラメータ名を `Keycloak*` → `Oidc*` にリネーム、audience をパラメータ化
- `frontend/src/config/oidc.ts`: 環境変数を `VITE_KEYCLOAK_*` → `VITE_OIDC_*` にリネーム、authority URL を直接指定
- LINE Login 統合は方式A（Cognito 外部 IdP）を推奨として併記

### 残課題
- LINE Login 統合方式の最終決定（実装フェーズ）
- 既存テストの環境変数名更新の詳細範囲
- `.env.example` 等の設定ファイルテンプレートの更新

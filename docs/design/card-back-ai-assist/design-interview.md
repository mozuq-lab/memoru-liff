# カード AI アシスト入力 設計ヒアリング記録

**作成日**: 2026-03-03
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存の AI サービス基盤と CardForm コンポーネントの設計を確認し、新機能の API 設計方針を決定するためのヒアリングを実施しました。

## 質問と回答

### Q1: API 設計方針（表面・裏面の処理方式）

**カテゴリ**: アーキテクチャ
**背景**: 表面の「表現整理」と裏面の「補足・拡充」で AI の役割が異なるため、1 回の API コールでまとめるか、別々に処理するか方針を決定する必要があった

**回答**: 1 回でまとめて処理 — 表面・裏面をまとめて AI に送信し、両方の結果を 1 レスポンスで返す。シンプルでレイテンシが低い。

**信頼性への影響**:
- API エンドポイント設計が 🟡 → 🔵 に確定（`POST /cards/refine` で表面・裏面を一括処理）
- プロンプト設計方針が確定（1 つのプロンプトで表面整理 + 裏面補足を指示）

---

## ヒアリング結果サマリー

### 確認できた事項
- 表面・裏面を 1 回の API コールでまとめて処理する
- 既存の AI サービスパターン（Protocol → Strands 実装）を踏襲する

### 設計方針の決定事項
- API エンドポイント: `POST /cards/refine`（1 エンドポイントで両方処理）
- プロンプト: 表面整理と裏面補足を 1 つのシステムプロンプトで定義
- レスポンス: `{ refined_front, refined_back }` で両方を返す

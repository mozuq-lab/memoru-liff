# Cognito LINE Login 外部 IdP 統合 ヒアリング記録

**作成日**: 2026-03-03
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

認証プロバイダ切り替え（`auth-provider-switch`）の設計文書で「設計フェーズで決定」とされていた LINE Login 統合方式（REQ-404）について、方式A の具体的な要件を確定するためのヒアリングを実施しました。

## 質問と回答

### Q1: LINE ユーザー ID 連携の方針

**カテゴリ**: 既存設計確認
**背景**: 現在 `/users/link-line` で手動リンクしている `line_user_id` を、Cognito フェデレーションで自動化するかの判断が必要。Cognito は LINE Login 経由ユーザーの LINE 属性を取得可能だが、DynamoDB の `line_user_id` フィールドへの自動反映には追加実装が必要。

**回答**: **後回し** — 今回は CDK 側の外部 IdP 設定のみ。LINE ユーザー ID 自動連携は別タスクで対応。既存の手動リンク機能（`/users/link-line`）はそのまま維持。

**信頼性への影響**:
- REQ-402（手動リンク維持）を 🔵 青信号で確定
- REQ-405（自動連携スコープ外）を 🔵 青信号で確定
- 実装スコープが CDK のみに限定され、フロントエンド・バックエンドの変更が不要であることが確定

---

### Q2: 認証方式の併存

**カテゴリ**: 追加要件
**背景**: Cognito Hosted UI で LINE Login のみにするか、メール+パスワード認証も残すかの判断が必要。LIFF 外（ブラウザ直接アクセス等）での利用や、開発者のデバッグ用途を考慮。

**回答**: **両方併存（推奨）** — LINE Login + メール認証の両方を有効にし、ユーザーが選択できるようにする。

**信頼性への影響**:
- REQ-005（両方の認証方式利用可能）を 🔵 青信号で新規追加
- REQ-004 の `supportedIdentityProviders` に Cognito + LINE の両方を含めることが確定

---

### Q3: Cognito Hosted UI の利用

**カテゴリ**: 未定義部分詳細化
**背景**: 方式A では Cognito Hosted UI へのリダイレクトが発生する。`oidc-client-ts` は標準 OIDC フローとして Hosted UI にリダイレクトするが、LIFF 内でのリダイレクト体験が許容可能かの確認が必要。

**回答**: **許容する** — `oidc-client-ts` が Cognito Hosted UI にリダイレクトし、そこで LINE Login ボタンを表示。コード変更最小限を優先。

**信頼性への影響**:
- REQ-403（フロントエンド変更不要）を 🔵 青信号で確定
- カスタム UI 開発のスコープ除外が確定

---

### Q4: コードベース詳細分析の要否

**カテゴリ**: 影響範囲
**背景**: CDK 以外のファイルへの影響を網羅的に調査するかの判断。

**回答**: **不要** — ここまでの調査で十分。設計文書のみで要件定義を作成。

**信頼性への影響**:
- 既に確認済みのファイル一覧（note.md 記載）に基づき、変更不要箇所を 🔵 青信号で確定

## ヒアリング結果サマリー

### 確認できた事項
- 方式A（Cognito 外部 OIDC IdP）で進めること
- LINE Login とメール+パスワード認証の両方を提供すること
- Cognito Hosted UI へのリダイレクトは許容すること
- LINE ユーザー ID の自動連携は本要件のスコープ外
- 実装スコープは CDK（`cognito-stack.ts` + `app.ts`）のみ

### 追加/変更要件
- REQ-005（両方の認証方式利用可能）を新規追加

### 残課題
- LINE Login の OIDC エンドポイントが Cognito の自動検出に対応しているか（実装時に確認）
- LINE Login チャネルの Channel Secret の安全な管理方法（環境変数 or Secrets Manager）

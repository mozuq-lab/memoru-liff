# review-undo ヒアリング記録

**作成日**: 2026-02-28
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存の復習機能（review-flow）の設計文書・実装を確認し、復習取り消し（undo）機能の要件を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: 取り消しの対象範囲

**質問日時**: 2026-02-28
**カテゴリ**: 未定義部分詳細化
**背景**: 復習セッションでは複数カードを順番に採点するため、取り消し対象を「直前の1枚のみ」にするか「セッション内の任意のカード」にするかで実装方式が異なる。

**回答**: セッション内の任意のカード

**信頼性への影響**:
- この回答により、REQ-005, REQ-401 の信頼性レベルが 🔴 → 🔵 に向上
- 完了画面に全カードの一覧を表示する設計が確定

---

### Q2: 取り消し後の動作

**質問日時**: 2026-02-28
**カテゴリ**: 未定義部分詳細化
**背景**: 取り消し後に「SRSパラメータを復元するだけ」か「復元＋再採点UIを提供する」かで、UIフローとAPI設計が異なる。

**回答**: 取り消し＋再採点（推奨）

**信頼性への影響**:
- この回答により、REQ-006, REQ-007, REQ-008 の信頼性レベルが 🔴 → 🔵 に向上
- Undo API + カード画面への遷移 + 再採点のフルフローが確定

---

### Q3: 完了画面に表示する情報

**質問日時**: 2026-02-28
**カテゴリ**: 未定義部分詳細化
**背景**: 現在の完了画面は復習枚数のみ表示している。カードごとの詳細情報をどこまで表示するかを決定する必要がある。

**回答**: カードの表面テキスト、採点結果（グレード）、次の復習日、取り消しボタン（全4項目を選択）

**信頼性への影響**:
- この回答により、REQ-001, REQ-002 の信頼性レベルが 🔴 → 🔵 に向上
- 完了画面の情報量が確定し、ReviewCompleteコンポーネントの大幅な拡張が必要と判明

---

### Q4: 取り消しのUIフロー

**質問日時**: 2026-02-28
**カテゴリ**: 未定義部分詳細化
**背景**: 取り消し＋再採点のUIフローとして、完了画面上で直接再採点する方式と、カード復習画面に戻る方式がある。

**回答**: カード画面に戻って再採点

**信頼性への影響**:
- この回答により、REQ-007, REQ-203 の信頼性レベルが 🔴 → 🔵 に向上
- 既存のFlipCard + GradeButtonsコンポーネントを再利用する方式が確定

---

### Q5: スキップしたカードの表示

**質問日時**: 2026-02-28
**カテゴリ**: 未定義部分詳細化
**背景**: セッション中にスキップしたカードを完了画面の結果一覧に含めるかどうかで、表示内容が変わる。

**回答**: 表示する（推奨）

**信頼性への影響**:
- この回答により、REQ-003, REQ-104 の信頼性レベルが 🔴 → 🔵 に向上
- スキップカードは「スキップ」ステータスで表示、取り消しボタンなしの設計が確定

---

### Q6: 取り消し時の履歴管理

**質問日時**: 2026-02-28
**カテゴリ**: 既存設計確認
**背景**: reviews テーブルのレコードを取り消し時に削除するかどうかで、データモデルの設計が異なる。既存のReviewResponseにはprevious状態が含まれており、復元に利用可能。

**回答**: 残す（推奨）

**信頼性への影響**:
- この回答により、REQ-011 の信頼性レベルが 🔴 → 🔵 に向上
- reviews テーブルは分析用途のため既存レコードを保持し、カードのSRSパラメータのみ復元する方式が確定

---

## ヒアリング結果サマリー

### 確認できた事項
- 取り消し対象: セッション内の任意のカード（直前のみではない）
- 取り消し後の動作: SRSパラメータ復元 + カード画面に遷移して再採点
- 完了画面の表示: カード表面テキスト、グレード、次の復習日、取り消しボタン
- スキップカードも結果一覧に含める（取り消しボタンは非表示）
- reviews テーブルのレコードは保持（削除しない）

### 追加/変更要件
- REQ-001〜REQ-014: 完了画面の結果表示・取り消し機能の新規要件
- REQ-101〜REQ-104: 条件付き要件の新規追加
- REQ-201〜REQ-203: 状態要件の新規追加
- 既存のReviewCompleteコンポーネントの大幅拡張が必要
- ReviewPageで採点結果をセッション中保持する仕組みが必要
- 新規APIエンドポイント `POST /reviews/{cardId}/undo` の追加

### 残課題
- Undo APIの詳細なリクエスト/レスポンス形式（設計フェーズで確定）
- 再採点時のUI遷移アニメーションの詳細（設計フェーズで確定）
- 同一カードの複数回取り消し時の挙動の詳細確認

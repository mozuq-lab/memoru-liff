# review-undo 要件定義書

## 概要

復習完了画面にカードごとの採点結果・次回復習日・取り消しボタンを表示し、セッション内の任意のカードの採点を取り消して再採点できる機能を実装する。バックエンドにUndoAPIを追加し、SRSパラメータを復元した上でカード画面に戻って再採点する方式を採用する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングにない推測による要件

### 通常要件

#### 完了画面の結果表示

- REQ-001: システムは復習完了画面に、セッション中に復習した全カードの一覧を表示しなければならない 🔵 *ユーザヒアリングより*
- REQ-002: 各カードの表示項目には、カード表面テキスト・採点結果（グレード0〜5）・次の復習日を含まなければならない 🔵 *ユーザヒアリングより*
- REQ-003: スキップしたカードもセッション結果一覧に表示しなければならない 🔵 *ユーザヒアリングより*
- REQ-004: スキップしたカードは「スキップ」と表示し、取り消しボタンは表示しない 🟡 *ユーザヒアリング「スキップも表示する」から妥当な推測*

#### 取り消し機能

- REQ-005: 各採点済みカードに取り消しボタンを表示しなければならない 🔵 *ユーザヒアリングより*
- REQ-006: 取り消しボタン押下時、バックエンドAPIを呼び出してSRSパラメータ（ease_factor, interval, repetitions, next_review_at）を採点前の状態に復元しなければならない 🔵 *ユーザヒアリング「Aの案（バックエンドUndo API）」より*
- REQ-007: SRSパラメータ復元後、該当カードの復習画面に遷移して再採点できるようにしなければならない 🔵 *ユーザヒアリング「カード画面に戻って再採点」より*
- REQ-008: 再採点完了後、完了画面に戻り結果一覧を更新しなければならない 🟡 *ユーザヒアリングから妥当な推測*

#### バックエンドAPI

- REQ-009: `POST /reviews/{cardId}/undo` エンドポイントを提供しなければならない 🟡 *バックエンドUndo APIの要件から妥当な推測*
- REQ-010: Undo APIはカードのSRSパラメータ（ease_factor, interval, repetitions, next_review_at）を前回の状態に復元しなければならない 🔵 *ユーザヒアリング・既存ReviewResponse.previousより*
- REQ-011: Undo APIはreviews テーブルの既存レコードを削除せず保持しなければならない 🔵 *ユーザヒアリング「残す」より*
- REQ-012: Undo APIはreview_historyの最新エントリを削除してSRSパラメータを復元しなければならない 🟡 *review_service.py既存実装から妥当な推測*

#### フロントエンド状態管理

- REQ-013: ReviewPageは各カードの採点結果（grade, ReviewResponse）をセッション中保持しなければならない 🟡 *既存ReviewPage実装から妥当な推測*
- REQ-014: ReviewCompleteコンポーネントはセッション結果一覧と取り消し操作を管理しなければならない 🟡 *既存ReviewComplete実装から妥当な推測*

### 条件付き要件

- REQ-101: 取り消しボタンが押された場合、システムはUndo APIを呼び出してからカード復習画面に遷移しなければならない 🔵 *ユーザヒアリングより*
- REQ-102: Undo API呼び出しが失敗した場合、システムはエラーメッセージを表示して完了画面に留まらなければならない 🟡 *一般的なエラーハンドリングから妥当な推測*
- REQ-103: 再採点が完了した場合、システムは結果一覧の該当カード情報を更新しなければならない 🟡 *ユーザヒアリングから妥当な推測*
- REQ-104: セッション中にカードがスキップされた場合、結果一覧では「スキップ」ステータスで表示しなければならない 🔵 *ユーザヒアリングより*

### 状態要件

- REQ-201: 完了画面にある場合、システムはセッション中の全採点結果を保持し表示しなければならない 🟡 *ユーザヒアリングから妥当な推測*
- REQ-202: 取り消し処理中にある場合、システムは取り消しボタンを無効化してローディング表示しなければならない 🟡 *一般的なUXパターンから妥当な推測*
- REQ-203: 再採点モードにある場合、システムは通常の復習フローと同じカード表示・採点UIを提供しなければならない 🔵 *ユーザヒアリング「カード画面に戻って再採点」より*

### オプション要件

- REQ-301: 完了画面の結果一覧はグレードに応じた色分け表示をしてもよい 🟡 *既存GradeButtonsの色分けから妥当な推測*
- REQ-302: 次の復習日は相対表示（「明日」「3日後」等）をしてもよい 🟡 *UXの一般的なパターンから妥当な推測*

### 制約要件

- REQ-401: 取り消し対象はセッション内で採点済みのカードのみとする 🔵 *ユーザヒアリング「セッション内の任意のカード」より*
- REQ-402: 取り消しはセッション中（完了画面表示中）のみ可能とし、ホームに戻った後は不可とする 🟡 *セッション管理の一般的な設計から妥当な推測*
- REQ-403: フロントエンドはReact + TypeScript + Tailwind CSSで実装する 🔵 *CLAUDE.md技術スタックより*
- REQ-404: バックエンドはPython 3.12 + AWS SAM (Lambda + API Gateway + DynamoDB)で実装する 🔵 *CLAUDE.md技術スタックより*
- REQ-405: 既存のreviews APIエンドポイントとの整合性を維持する 🔵 *既存API設計より*

## 非機能要件

### パフォーマンス

- NFR-001: Undo API のレスポンス時間は500ms以内とする 🟡 *既存APIパフォーマンス基準から妥当な推測*
- NFR-002: 完了画面の結果一覧表示はカード50枚まで1秒以内に描画する 🟡 *一般的なWeb UIパフォーマンス基準から妥当な推測*

### セキュリティ

- NFR-101: Undo APIは認証済みユーザーのみアクセス可能とする 🔵 *既存API認証設計より*
- NFR-102: ユーザーは自身のカードのみ取り消し可能とする 🔵 *既存API認可設計より*

### ユーザビリティ

- NFR-201: 取り消しボタンは最小タッチターゲット44x44pxを確保する 🔵 *既存review-flow NFR-201より*
- NFR-202: 完了画面は復習枚数に関わらず適切にスクロール表示する 🟡 *一般的なモバイルUXから妥当な推測*
- NFR-203: グレード表示は既存のGradeButtonsコンポーネントの色スキームと統一する 🟡 *UIの一貫性から妥当な推測*

## Edgeケース

### エラー処理

- EDGE-001: Undo API呼び出し中にネットワークエラーが発生した場合、エラーを表示して完了画面に留まる 🟡 *一般的なエラーハンドリングから妥当な推測*
- EDGE-002: 取り消し後の再採点中にエラーが発生した場合、カードは取り消し済み（未採点）状態のまま完了画面に戻る 🟡 *データ整合性の観点から妥当な推測*
- EDGE-003: 同一カードに対して複数回の取り消し・再採点が行われた場合、最新の状態を正しく反映する 🟡 *データ整合性の観点から妥当な推測*

### 境界値

- EDGE-101: セッション中のカードが1枚のみの場合でも、結果一覧と取り消しが正常に動作する 🔵 *ユーザヒアリング（1枚カードのケースが議論の起点）より*
- EDGE-102: セッション中の全カードがスキップされた場合、結果一覧にスキップ表示のみが並ぶ 🟡 *スキップ仕様から妥当な推測*
- EDGE-103: カード表面テキストが長い場合、適切に省略表示する 🟡 *一般的なUI設計から妥当な推測*

## REQ-001〜002: 完了画面の結果一覧表示 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

### Given（前提条件）
- ユーザーが復習セッションを完了している
- セッション中に少なくとも1枚のカードが採点されている

### When（実行条件）
- 完了画面が表示される

### Then（期待結果）
- セッション中の全カードが一覧表示される
- 各カードにカード表面テキスト、採点結果（グレード）、次の復習日が表示される

### テストケース

#### 正常系

- [ ] **TC-001-01**: 採点済みカード1枚の結果表示 🔵
  - **入力**: グレード4で1枚採点済みのセッション
  - **期待結果**: カード表面テキスト、「4」のグレード表示、次回復習日が表示される
  - **信頼性**: 🔵 *ユーザヒアリング（1枚カードケースが起点）より*

- [ ] **TC-001-02**: 複数カードの結果表示 🔵
  - **入力**: 5枚のカードを異なるグレードで採点済みのセッション
  - **期待結果**: 5枚すべてのカードが一覧表示され、それぞれのグレードと次回復習日が正しい
  - **信頼性**: 🔵 *ユーザヒアリングより*

- [ ] **TC-001-03**: カード表面テキストの表示 🟡
  - **入力**: 表面テキストが100文字のカード
  - **期待結果**: テキストが適切に省略表示される（末尾「...」等）
  - **信頼性**: 🟡 *一般的なUI設計から妥当な推測*

- [ ] **TC-001-04**: 次回復習日の表示 🟡
  - **入力**: グレード5で採点したカード（interval=6日）
  - **期待結果**: 次回復習日が正しい日付で表示される
  - **信頼性**: 🟡 *SM-2アルゴリズムの計算結果から妥当な推測*

## REQ-005〜008: 取り消しと再採点 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

### Given（前提条件）
- 完了画面が表示されている
- 採点済みカードに取り消しボタンが表示されている

### When（実行条件）
- 取り消しボタンを押す

### Then（期待結果）
- Undo APIが呼び出され、SRSパラメータが復元される
- カード復習画面に遷移する
- 再採点完了後、完了画面に戻り結果が更新される

### テストケース

#### 正常系

- [ ] **TC-003-01**: 取り消しボタン押下でAPI呼び出し 🔵
  - **入力**: 採点済みカードの取り消しボタンを押下
  - **期待結果**: `POST /reviews/{cardId}/undo` が呼び出される
  - **信頼性**: 🔵 *ユーザヒアリングより*

- [ ] **TC-003-02**: 取り消し後のカード画面遷移 🔵
  - **入力**: Undo API成功レスポンス
  - **期待結果**: 該当カードの復習画面に遷移し、カード表面が表示される
  - **信頼性**: 🔵 *ユーザヒアリング「カード画面に戻って再採点」より*

- [ ] **TC-003-03**: 再採点後の完了画面復帰 🟡
  - **入力**: 再採点でグレード5を選択
  - **期待結果**: 完了画面に戻り、該当カードのグレードが5に更新、次回復習日も更新
  - **信頼性**: 🟡 *ユーザヒアリングから妥当な推測*

- [ ] **TC-003-04**: 取り消し中のローディング表示 🟡
  - **入力**: 取り消しボタン押下
  - **期待結果**: API呼び出し中は取り消しボタンが無効化され、ローディング表示
  - **信頼性**: 🟡 *一般的なUXパターンから妥当な推測*

- [ ] **TC-003-05**: 同一カードの複数回取り消し 🟡
  - **入力**: カードAを取り消し→再採点→再度取り消し→再採点
  - **期待結果**: 2回目の取り消し・再採点も正常に動作し、最新の結果が表示される
  - **信頼性**: 🟡 *データ整合性の観点から妥当な推測*

#### 異常系

- [ ] **TC-003-E01**: 取り消しAPI失敗 🟡
  - **入力**: ネットワークエラーでUndo APIが失敗
  - **期待結果**: エラーメッセージ表示、完了画面に留まる、結果一覧は変更なし
  - **信頼性**: 🟡 *一般的なエラーハンドリングから妥当な推測*

- [ ] **TC-003-E02**: 再採点API失敗 🟡
  - **入力**: 取り消し成功後、再採点のsubmitReviewが失敗
  - **期待結果**: カードは取り消し済み（未採点）状態で完了画面に戻る
  - **信頼性**: 🟡 *データ整合性の観点から妥当な推測*

## REQ-009〜012: Undo API 🔵

**信頼性**: 🔵 *ユーザヒアリング・既存review_service.py実装より*

### Given（前提条件）
- 認証済みユーザーがリクエストを送信
- 対象カードにreview_historyが存在する

### When（実行条件）
- `POST /reviews/{cardId}/undo` にリクエスト

### Then（期待結果）
- SRSパラメータが前回の状態に復元される
- review_historyの最新エントリが削除される
- reviews テーブルのレコードは保持される
- 復元後の状態がレスポンスとして返される

### テストケース

#### 正常系

- [ ] **TC-004-01**: SRSパラメータの復元 🔵
  - **入力**: ease_factor=2.5, interval=10, repetitions=3 のカードをグレード0で採点後にundo
  - **期待結果**: ease_factor, interval, repetitions, next_review_at が採点前の値に戻る
  - **信頼性**: 🔵 *既存ReviewResponse.previousの構造より*

- [ ] **TC-004-02**: review_historyの最新エントリ削除 🟡
  - **入力**: review_historyに3件ある状態でundo
  - **期待結果**: review_historyが2件になる
  - **信頼性**: 🟡 *review_service.py既存実装から妥当な推測*

- [ ] **TC-004-03**: reviewsテーブルレコード保持 🔵
  - **入力**: undo実行
  - **期待結果**: reviews テーブルのレコードは削除されず保持される
  - **信頼性**: 🔵 *ユーザヒアリング「残す」より*

- [ ] **TC-004-04**: Undoレスポンスの返却 🟡
  - **入力**: undo実行
  - **期待結果**: 復元後のSRSパラメータがレスポンスに含まれる
  - **信頼性**: 🟡 *API設計の一般的なパターンから妥当な推測*

#### 異常系

- [ ] **TC-004-E01**: 未認証ユーザーのアクセス 🔵
  - **入力**: 認証なしでUndo APIにアクセス
  - **期待結果**: 401 Unauthorized
  - **信頼性**: 🔵 *既存API認証設計より*

- [ ] **TC-004-E02**: 他ユーザーのカードのundo 🔵
  - **入力**: 別ユーザーのcardIdを指定
  - **期待結果**: 404 Not Found
  - **信頼性**: 🔵 *既存API認可設計より*

- [ ] **TC-004-E03**: review_historyが空のカードのundo 🟡
  - **入力**: 一度も復習されていないカードのcardId
  - **期待結果**: 400 Bad Request (取り消し対象なし)
  - **信頼性**: 🟡 *データ整合性の観点から妥当な推測*

- [ ] **TC-004-E04**: 存在しないカードIDのundo 🔵
  - **入力**: 存在しないcardId
  - **期待結果**: 404 Not Found
  - **信頼性**: 🔵 *既存API設計パターンより*

# 復習機能の実装 ユーザストーリー

**作成日**: 2026-02-25
**関連要件定義**: [requirements.md](requirements.md)
**ヒアリング記録**: [interview-record.md](interview-record.md)

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングを参考にした確実なストーリー
- 🟡 **黄信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングから妥当な推測によるストーリー
- 🔴 **赤信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングにない推測によるストーリー

---

## エピック1: 復習セッション

### ストーリー 1.1: 復習を開始する 🔵

**信頼性**: 🔵 *ユーザヒアリング・既存API仕様より*

**私は** 復習したいユーザー **として**
**ホーム画面の「復習を始める」ボタンをタップして復習セッションを開始したい**
**そうすることで** 期限が来たカードを効率的に復習できる

**関連要件**: REQ-RV-030, REQ-RV-040

**詳細シナリオ**:
1. ユーザーがホーム画面の「復習を始める」ボタンをタップ
2. `/review` ページに遷移
3. `GET /cards/due` で復習対象カードを取得
4. ローディング表示後、最初のカードの表面が表示される
5. 進捗バーに「1 / N」が表示される

**前提条件**:
- ユーザーがログイン済み
- 復習対象カードが1枚以上ある

**制約事項**:
- ナビゲーションバーは非表示
- 「戻る」ボタンで中断可能

**優先度**: Must Have

---

### ストーリー 1.2: カードをフリップして答えを確認する 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

**私は** 復習中のユーザー **として**
**カードをタップして裏面（解答）を確認したい**
**そうすることで** 自分の回答が正しかったか確認できる

**関連要件**: REQ-RV-002, REQ-RV-201, REQ-RV-202

**詳細シナリオ**:
1. カードの表面（質問）が表示されている
2. ユーザーが心の中で回答を考える
3. カードエリアをタップする
4. フリップアニメーションでカードが裏返る
5. 裏面（解答）が表示される
6. 採点ボタン（0-5）とスキップボタンが表示される

**前提条件**:
- 復習セッションが開始済み
- カードが表面表示状態

**優先度**: Must Have

---

### ストーリー 1.3: 自己採点する 🔵

**信頼性**: 🔵 *ユーザヒアリング・既存API仕様 REQ-032より*

**私は** 復習中のユーザー **として**
**覚えていた度合いを0-5で採点したい**
**そうすることで** SM-2アルゴリズムが次回の復習間隔を最適化してくれる

**関連要件**: REQ-RV-003, REQ-RV-041

**詳細シナリオ**:
1. カードの裏面が表示され、採点ボタンが表示されている
2. ユーザーが0-5のいずれかのボタンをタップ
3. `POST /reviews/{card_id}` で採点結果を送信
4. 次のカードの表面が表示される（進捗バー更新）
5. 最後のカードの場合は復習完了画面に遷移

**前提条件**:
- カードが裏面表示状態
- 採点ボタンが表示されている

**制約事項**:
- grade は 0-5 の整数（バックエンドバリデーション済み）
- 0: 全く覚えていない / 5: 完璧

**優先度**: Must Have

---

### ストーリー 1.4: カードをスキップする 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

**私は** 復習中のユーザー **として**
**今は復習したくないカードをスキップしたい**
**そうすることで** 他のカードの復習に集中できる

**関連要件**: REQ-RV-020, REQ-RV-021

**詳細シナリオ**:
1. カードの裏面が表示されている（または表面状態でも可）
2. ユーザーが「スキップ」ボタンをタップ
3. SRSパラメータは更新されない（APIリクエストなし）
4. 次のカードの表面が表示される
5. 進捗バーが更新される

**前提条件**:
- 復習セッション中

**優先度**: Should Have

---

## エピック2: 復習完了

### ストーリー 2.1: 復習完了を確認する 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

**私は** 復習を終えたユーザー **として**
**復習が完了したことと復習枚数を確認したい**
**そうすることで** 達成感を得て次の学習に繋げられる

**関連要件**: REQ-RV-010, REQ-RV-011

**詳細シナリオ**:
1. 最後のカードの採点が完了（またはすべてスキップ）
2. 復習完了画面が表示される
3. 「復習完了！」メッセージと復習した枚数が表示される
4. 「ホームに戻る」ボタンが表示される
5. ボタンタップでホーム画面に遷移

**前提条件**:
- すべてのカードが採点済みまたはスキップ済み

**優先度**: Must Have

---

## エピック3: ナビゲーション連携

### ストーリー 3.1: カード一覧から復習を開始する 🔵

**信頼性**: 🔵 *ユーザヒアリングより*

**私は** カード一覧を見ているユーザー **として**
**復習対象タブから直接復習を開始したい**
**そうすることで** 復習対象のカードを確認してから復習に入れる

**関連要件**: REQ-RV-031

**詳細シナリオ**:
1. ユーザーがカード一覧画面の「復習対象」タブを表示している
2. 復習対象カードがある場合、「復習開始」ボタンが表示される
3. ボタンをタップすると `/review` に遷移
4. 復習セッションが開始される

**前提条件**:
- カード一覧画面の「復習対象」タブが表示中
- 復習対象カードが1枚以上ある

**優先度**: Should Have

---

### ストーリー 3.2: 復習セッションを中断する 🟡

**信頼性**: 🟡 *ユーザヒアリングから妥当な推測*

**私は** 復習中に他の用事ができたユーザー **として**
**復習セッションを中断して画面を離れたい**
**そうすることで** 後から残りのカードを復習できる

**関連要件**: REQ-RV-402

**詳細シナリオ**:
1. 復習セッション中に「戻る」ボタンをタップ
2. 前の画面（ホームまたはカード一覧）に遷移
3. 採点済みのカードは保存済み（API送信済み）
4. 未採点のカードは復習対象のまま残る

**前提条件**:
- 復習セッション中

**備考**: 中断時の確認ダイアログは不要（採点済みカードは既にAPI送信済みのため）

**優先度**: Must Have

---

## エピック4: エラー処理

### ストーリー 4.1: 復習対象カードがない場合 🟡

**信頼性**: 🟡 *既存実装パターンから妥当な推測*

**私は** 復習画面にアクセスしたユーザー **として**
**復習対象カードがない場合、その旨を確認したい**
**そうすることで** 不要な操作を避けられる

**関連要件**: REQ-RV-101

**詳細シナリオ**:
1. `/review` にアクセス
2. `GET /cards/due` で復習対象カードを取得
3. 復習対象が0枚の場合、空状態メッセージを表示
4. 「ホームに戻る」ボタンを表示

**優先度**: Must Have

---

### ストーリー 4.2: 採点送信に失敗した場合 🟡

**信頼性**: 🟡 *既存実装パターンから妥当な推測*

**私は** 復習中に通信エラーが発生したユーザー **として**
**エラー内容を確認して再送信したい**
**そうすることで** 採点結果が失われない

**関連要件**: REQ-RV-104

**詳細シナリオ**:
1. 採点ボタンをタップ
2. `POST /reviews/{card_id}` が失敗
3. エラーメッセージを表示
4. 「再送信」ボタンで同じ採点を再送信可能

**優先度**: Should Have

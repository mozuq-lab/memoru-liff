# 認証プロバイダ切り替え ヒアリング記録

**作成日**: 2026-03-02
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

認証プロバイダ（Keycloak / Cognito）切り替え機能の要件を明確化するため、既存の設計文書・実装コードの調査結果を踏まえ、ユーザーに不明点・方針を確認しました。

## 質問と回答

### Q1: 切り替え対象のプロバイダ

**カテゴリ**: 未定義部分詳細化
**背景**: 切り替え先として Cognito 以外（Auth0、LIFFトークン直接検証等）も候補になり得るため、スコープを確定する必要があった。

**回答**: Keycloak（現状維持）と Amazon Cognito の2つ。LIFFトークン直接検証は対象外。

**信頼性への影響**:
- REQ-001〜REQ-004 の信頼性レベルが 🟡 → 🔵 に向上（対象プロバイダが確定）

---

### Q2: 切り替え粒度

**カテゴリ**: 未定義部分詳細化
**背景**: ランタイム切り替え（同一デプロイ内で動的に変更）か、デプロイ時切り替え（環境変数/パラメータ変更 + 再デプロイ）かで設計の複雑度が大きく異なるため確認。

**回答**: 環境変数のみ（デプロイ時の設定変更で切り替え）。コード変更不要であること。

**信頼性への影響**:
- REQ-003, REQ-004 の信頼性レベルが 🔵 に確定（コード変更不要の方針が明確化）
- ランタイム切り替えの複雑な要件（プロバイダ判定ロジック、マルチ Authorizer 等）が不要であることが確定

---

### Q3: コードベース詳細分析の要否

**カテゴリ**: 既存設計確認
**背景**: これまでの会話で認証周りの実装を十分に調査済みだが、追加分析が必要か確認。

**回答**: 不要。これまでの会話で充分に把握済み。

**信頼性への影響**:
- 既存実装に関する知見（`shared.py`、`oidc.ts`、`template.yaml`）がヒアリングで裏付けられた

---

### Q4: Cognito 使用時の LINE Login 統合方式

**カテゴリ**: 未定義部分詳細化
**背景**: Cognito で LINE Login を使う場合、(a) Cognito 外部 IdP として設定、(b) LIFF トークン → Cognito カスタム認証フローの2方式がある。方式によりフロントエンドの実装が変わるため確認。

**回答**: まだ決めていない。要件定義では両方の選択肢を記載し、設計フェーズで決定する。

**信頼性への影響**:
- REQ-404 を新規追加（LINE Login 統合方式は設計フェーズで決定、信頼性: 🔵）
- Cognito 側の具体的な設定要件は 🟡（設計フェーズ依存）

---

## ヒアリング結果サマリー

### 確認できた事項
- 対象プロバイダは Keycloak と Cognito の2つに限定
- 切り替えはデプロイ時の環境変数/SAMパラメータ変更のみ
- コード変更は不要とする方針
- 既存の `oidc-client-ts` + API Gateway JWT Authorizer の構成を維持
- データ移行は別途対応（スコープ外）

### 追加/変更要件
- REQ-404: LINE Login 統合方式の決定を設計フェーズに委ねる
- REQ-405: データ移行をスコープ外として明示

### 残課題
- Cognito + LINE Login の具体的な統合方式（設計フェーズで決定）
- Cognito User Pool の audience 設定値
- 既存テストの Keycloak 固有テストデータの扱い

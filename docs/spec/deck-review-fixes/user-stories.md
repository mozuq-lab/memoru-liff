## エピック1: デッキ別カード表示の修正（High）

### ストーリー 1.1: デッキ別カード一覧 🔵

**信頼性**: 🔵 *レビュー H-1・要件 4.3 より*

**私は** 学習者 **として**
**デッキをタップしてそのデッキのカードだけを一覧で見たい**
**そうすることで** 特定のデッキに集中して学習管理ができる

**関連要件**: REQ-001, REQ-101, REQ-102

**詳細シナリオ**:
1. DecksPage でデッキをタップ
2. `/cards?deck_id=xxx` に遷移
3. CardsPage が `deck_id` クエリパラメータを読み取り
4. 該当デッキのカードのみが表示される
5. ページヘッダーにデッキ名が表示される

**前提条件**:
- デッキが作成済みで、カードが紐付いている
- バックエンド `GET /cards?deck_id=xxx` は対応済み

**優先度**: Must Have

## エピック2: カードのデッキ解除（High）

### ストーリー 2.1: カードを未分類に戻す 🔵

**信頼性**: 🔵 *レビュー H-2 より*

**私は** 学習者 **として**
**デッキに割り当てたカードを「未分類」に戻したい**
**そうすることで** カードの整理を柔軟に行える

**関連要件**: REQ-002, REQ-103, REQ-104

**詳細シナリオ**:
1. CardDetailPage でデッキ割り当て済みカードを表示
2. DeckSelector で「未分類」を選択
3. `deck_id: null` として API に送信
4. バックエンドが DynamoDB から `deck_id` 属性を REMOVE
5. カードが未分類状態に戻る

**前提条件**:
- カードが既にデッキに割り当てられている

**優先度**: Must Have

## エピック3: デッキ作成の堅牢性（High）

### ストーリー 3.1: デッキ数上限の正確なエラー通知 🔵

**信頼性**: 🔵 *追加レビュー H-3 より*

**私は** 学習者 **として**
**デッキ数上限に達した場合、適切なエラーメッセージを受け取りたい**
**そうすることで** なぜデッキを作成できないのか理解できる

**関連要件**: REQ-003

**詳細シナリオ**:
1. 50個のデッキが既に存在する状態
2. 新しいデッキを作成しようとする
3. API が HTTP 409 Conflict を返す
4. フロントエンドが適切なエラーメッセージを表示

**優先度**: Must Have

---

### ストーリー 3.2: 同時デッキ作成の競合防止 🔵

**信頼性**: 🔵 *追加レビュー H-4・ヒアリング回答より*

**私は** システム管理者 **として**
**同時にデッキ作成リクエストが来ても、制限を超えるデッキが作成されないことを保証したい**
**そうすることで** データの一貫性が保たれる

**関連要件**: REQ-004

**詳細シナリオ**:
1. ユーザーが49個のデッキを持つ状態
2. 2つの並行リクエストでデッキ作成を試みる
3. DynamoDB の ConditionExpression により1つのみが成功
4. もう1つは ConditionalCheckFailedException で失敗
5. HTTP 409 Conflict が返される

**前提条件**:
- DynamoDB の ConditionExpression が正しく設定されている

**優先度**: Must Have

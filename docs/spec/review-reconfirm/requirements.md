# review-reconfirm 要件定義書

## 概要

復習セッション中にquality 0-2（想起失敗）で評価されたカードを、セッション内の再確認ループに追加し、ユーザーが「覚えた」と言い切れる状態になるまで繰り返し表示する機能。セッション内完結（A案）で実装し、SM-2アルゴリズムは変更しない。

## 背景

現状、quality 0-2を選択してもカードは即座にセッションから外れ、翌日以降に再登場する。ユーザーが「覚えた」と確認しないまま学習が終わるため、その日の学習効果が不十分になる問題がある。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: ユーザヒアリングで明確に合意した要件
- 🟡 **黄信号**: ユーザヒアリング・既存実装から妥当な推測による要件
- 🔴 **赤信号**: ユーザヒアリング・既存実装にない推測による要件

### 通常要件

- REQ-001: 復習でquality 0-2が選択された場合、システムはSM-2計算を実行した後、そのカードをセッション内再確認キューの末尾に追加しなければならない 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-002: 再確認キューのカードが表示される場合、システムは「覚えた」「覚えていない」の2択のみを表示しなければならない 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-003: 再確認で「覚えた」が選択された場合、システムはそのカードをセッションから除外しなければならない（SM-2再計算・API呼び出しなし） 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-004: 再確認で「覚えていない」が選択された場合、システムはそのカードをキュー末尾に再追加しなければならない（API呼び出しなし） 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-005: 再確認カードの表示フローは「表面を見る → 裏面を確認 → 覚えた/覚えていない」でなければならない 🔵 *ユーザヒアリング 2026-02-28 確認*

### 条件付き要件

- REQ-101: 再確認モードでカードが表示される場合、システムは「再確認」バッジを表示しなければならない 🔵 *ヒアリング Q1 回答*
- REQ-102: 再確認モードで表示中の場合、システムは「スキップ」ボタンを非表示にしなければならない 🔵 *ヒアリング Q2 回答*
- REQ-103: quality 3-5が選択された場合、システムは従来通りカードをセッションから除外しなければならない（再確認ループに入らない） 🟡 *既存動作の維持、妥当な推測*

### 状態要件

- REQ-201: 再確認ループの状態はセッション内のフロントエンド状態のみで管理し、セッション終了時にリセットされなければならない 🔵 *ユーザヒアリング A案合意*
- REQ-202: 「覚えていない」が選択された場合、システムは復習日（next_review_at）を更新してはならない 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-203: 「覚えた」が選択された場合、システムは復習日を更新してはならない（SM-2が最初のquality 0-2評価時に既に設定済み） 🔵 *ユーザヒアリング 2026-02-28 合意*

### 制約要件

- REQ-401: SM-2アルゴリズムの計算ロジックを変更してはならない 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-402: 再確認ループの回数に上限を設けてはならない（ユーザーが「覚えた」を選択するまで繰り返す） 🔵 *ユーザヒアリング 2026-02-28 合意*
- REQ-403: バックエンドAPIの変更を行ってはならない（フロントエンドのみの変更で実現する） 🟡 *A案の設計方針から妥当な推測*
- REQ-404: 既存の取り消し（Undo）機能は再確認ループのカードに対しても使用可能でなければならない 🔵 *ヒアリング Q4 回答*

### UI要件

- REQ-501: 復習完了画面で再確認カードは元の評価（quality値）と再確認結果（「覚えた✔」）を両方表示しなければならない 🔵 *ヒアリング Q3 回答*
- REQ-502: 通常カードと再確認カードは同一キューで順番に流れなければならない（フェーズ分離メッセージなし） 🔵 *ヒアリング Q5 回答*

## 非機能要件

### パフォーマンス

- NFR-001: 再確認ループの状態管理はフロントエンドのメモリ内で完結し、追加のAPI呼び出しを発生させてはならない 🔵 *A案の設計方針*
- NFR-002: 再確認カードの表示切替は通常カードと同等の応答速度でなければならない 🟡 *UX品質の妥当な推測*

### ユーザビリティ

- NFR-201: 「覚えた」「覚えていない」ボタンはモバイル操作に適したサイズ（44px以上のタップ領域）でなければならない 🟡 *LINE LIFFモバイル対応の既存パターンから妥当な推測*
- NFR-202: 再確認バッジは通常の復習と明確に区別できるデザインでなければならない 🟡 *UX品質の妥当な推測*

## Edgeケース

### エラー処理

- EDGE-001: セッション中にアプリが閉じられた場合、再確認ループの状態は失われるが、SM-2がquality 0-2でinterval=1を設定済みのため翌日に再表示される 🔵 *SM-2動作とA案設計から確実*

### 境界値

- EDGE-101: 全カードがquality 0-2の場合、セッションは再確認カードのみのループとなる（正常動作） 🟡 *設計から妥当な推測*
- EDGE-102: 1枚のカードが複数回「覚えていない」→キュー末尾に再追加を繰り返す場合、無制限にループする（仕様通り） 🔵 *ユーザヒアリング合意*

### Undo連携

- EDGE-201: 復習完了画面でquality 0-2のカードを取り消した場合、SM-2結果が復元され、再確認ループからも除外される 🟡 *既存Undo機能との整合性から妥当な推測*
- EDGE-202: 取り消し後に再評価（regrade）でquality 0-2を選んだ場合、再び再確認ループに入る 🟡 *既存regrade機能との整合性から妥当な推測*
- EDGE-203: 取り消し後に再評価でquality 3-5を選んだ場合、再確認ループに入らない 🟡 *既存regrade機能との整合性から妥当な推測*

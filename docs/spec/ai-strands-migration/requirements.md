# ai-strands-migration 要件定義書

## 概要

Memoru LIFF の AI 機能を現行の boto3 直接呼び出し（Amazon Bedrock Claude 3 Haiku）から AWS Strands Agents SDK ベースの実装へ段階的に移行する。移行と同時に、回答採点/AI 評価・学習アドバイスの新機能を追加し、エージェントツール統合（Web 検索・URL 読み取り・画像解析）の拡張基盤を構築する。ローカル開発環境では Ollama プロバイダーにより Bedrock なしで AI 機能を動作確認できるようにする。

## 機能要件（EARS 記法）

**【信頼性レベル凡例】**:

- 🔵 **青信号**: ユーザヒアリング・既存実装コード・設計文書から確実に特定された要件
- 🟡 **黄信号**: ユーザヒアリング・技術仕様から妥当な推測による要件
- 🔴 **赤信号**: ヒアリング・設計文書にない推測による要件

---

### 通常要件

#### Strands Agents SDK 統合

- REQ-SM-001: システムは AWS Strands Agents SDK を AI サービスレイヤーとして統合しなければならない 🔵 *ユーザヒアリングより。現行 boto3 直接呼び出しから Strands Agents SDK ベースの実装へ移行する方針が確定*
- REQ-SM-002: システムは 既存のカード生成機能を Strands Agents 実装へ移行し、`generate_cards()` インターフェースを維持しなければならない 🔵 *ユーザヒアリングより。既存 API 契約（GenerateCardsResponse）の後方互換性を維持*

#### 新機能: 回答採点/AI 評価

- REQ-SM-003: システムは ユーザーの回答に対して AI による採点・評価機能を提供しなければならない 🔵 *ユーザヒアリングより。複数の新 AI 機能のうちの 1 つとして明確に要求された*

#### 新機能: 学習アドバイス

- REQ-SM-004: システムは ユーザーの学習履歴に基づいた AI による学習アドバイス機能を提供しなければならない 🔵 *ユーザヒアリングより。複数の新 AI 機能のうちの 1 つとして明確に要求された*

#### マルチモデルサポート

- REQ-SM-005: システムは Strands Agents のプロバイダー機構を使用して複数の AI モデル（Bedrock、Ollama）をサポートしなければならない 🔵 *ユーザヒアリングより。ローカル開発での Ollama 統合が明確に要求された*

#### エージェントツール統合

- REQ-SM-006: システムは Strands Agents の `@tool` デコレーターを使用したエージェントツール統合フレームワークを提供しなければならない 🔵 *ユーザヒアリングより。外部ツール（Web 検索、画像解析、URL 読み取り）の統合が要求された*

---

### 条件付き要件

#### 環境別プロバイダー切替

- REQ-SM-101: `ENVIRONMENT=dev` の場合、システムは Ollama プロバイダーを使用してローカル AI 実行を行わなければならない 🔵 *ユーザヒアリングより。完全ローカル環境での AI 動作確認が明確に要求された*

#### フィーチャーフラグ制御

- REQ-SM-102: `USE_STRANDS=true`（フィーチャーフラグ）の場合、システムは Strands Agents 実装を使用しなければならない 🔵 *ユーザヒアリングより。段階的移行のためのフィーチャーフラグ切替方式が確定*
- REQ-SM-103: `USE_STRANDS=false`（フィーチャーフラグ）の場合、システムは 既存の boto3 実装にフォールバックしなければならない 🔵 *ユーザヒアリングより。移行期間中の安全なフォールバックが要求された*

#### ツール実行エラー処理

- REQ-SM-104: エージェントツール実行が失敗した場合、システムは ツール結果なしでグレースフルデグレードしなければならない 🟡 *ユーザヒアリングのツール統合要件から妥当な推測。エージェントのマルチステップ推論ではツール失敗が発生しうる*

---

### 状態要件

#### 段階的移行フェーズ

- REQ-SM-201: 段階的移行フェーズにおいて、システムは 旧実装（boto3）と新実装（Strands Agents）の両方が共存可能でなければならない 🔵 *ユーザヒアリングより。フィーチャーフラグによる段階的移行方式が確定*

#### エージェント処理中の進捗通知

- REQ-SM-202: Strands Agent がマルチステップ推論を処理中の場合、システムは 進捗フィードバックを提供すべきである 🟡 *ユーザヒアリングの UX 要件から妥当な推測。エージェントの複合推論は処理時間が長くなる可能性がある*

---

### オプション要件

#### Web 検索ツール

- REQ-SM-301: システムは Web 検索ツールを統合し、カード生成の精度を向上させてもよい 🔵 *ユーザヒアリングより。外部ツールとして Web 検索が明確に挙げられた*

#### URL 読み取りツール

- REQ-SM-302: システムは URL 読み取りツールを統合し、ソースマテリアルの直接解析を可能にしてもよい 🔵 *ユーザヒアリングより。外部ツールとして URL 読み取りが明確に挙げられた*

#### 画像解析ツール

- REQ-SM-303: システムは 画像解析ツールを統合し、ビジュアル学習教材からのカード生成を可能にしてもよい 🔵 *ユーザヒアリングより。外部ツールとして画像解析が明確に挙げられた*

---

### 制約要件

#### Lambda タイムアウト設定

- REQ-SM-401: システムは Lambda タイムアウトを機能別に設定しなければならない（カード生成: 30 秒、回答採点: 60 秒、学習アドバイス: 60 秒） 🔵 *ユーザヒアリングより。機能別タイムアウト値が明確に指定された*

#### API レスポンス形式互換性

- REQ-SM-402: システムは 既存の API レスポンス形式（`GenerateCardsResponse`）を維持しなければならない 🔵 *既存実装（`backend/src/models/generate.py`）より。フロントエンドとの API 契約を破壊してはならない*

#### Pydantic v2 使用

- REQ-SM-403: システムは すべての新規データモデルに Pydantic v2 を使用しなければならない 🔵 *CLAUDE.md 技術スタック定義および既存実装の一貫性より*

#### テストカバレッジ維持

- REQ-SM-404: システムは テストカバレッジ 80% 以上を維持しなければならない 🔵 *CLAUDE.md 注意事項より。プロジェクト全体のテスト要件*

#### 既存テスト保護

- REQ-SM-405: システムは 既存の 260 件以上のバックエンドテストを破壊してはならない 🔵 *CLAUDE.md・既存テストスイートより。移行によるリグレッションを防止*

## 非機能要件

### パフォーマンス

- NFR-SM-001: カード生成のレスポンスタイムは 30 秒以内でなければならない（現行ベースライン維持） 🔵 *既存実装（`BedrockService.DEFAULT_TIMEOUT = 30`）および `template.yaml` Globals Timeout: 30 より*
- NFR-SM-002: 回答採点のレスポンスタイムは 10 秒以内であるべき 🟡 *ユーザヒアリングの機能別タイムアウト要件から妥当な推測。採点は単純な推論タスクのため短時間を期待*
- NFR-SM-003: 学習アドバイスのレスポンスタイムは 15 秒以内であるべき 🟡 *ユーザヒアリングの機能別タイムアウト要件から妥当な推測。アドバイス生成は中程度の推論タスク*

### 依存管理

- NFR-SM-004: Strands Agents SDK のバージョンは `requirements.txt` でピン留めしなければならない 🟡 *SDK 安定性への懸念から妥当な推測。初期段階の SDK では破壊的変更のリスクがある*

### セキュリティ

- NFR-SM-101: IAM 権限は最小必要権限に限定しなければならない（`bedrock:InvokeModel`、`bedrock:InvokeModelWithResponseStream`） 🔵 *既存 `template.yaml` の IAM ポリシー設計方針および AWS セキュリティベストプラクティスより*
- NFR-SM-102: ユーザー入力はプロンプトインジェクション防止のためにサニタイズしなければならない 🟡 *note.md セキュリティ注意事項から妥当な推測。エージェント機能拡張に伴いプロンプトインジェクションリスクが増大*

### 運用監視

- NFR-SM-201: システムは AWS Lambda Powertools による構造化ロギングを使用しなければならない 🔵 *既存実装および CLAUDE.md 技術スタックより。プロジェクト全体の標準ロギング方式*
- NFR-SM-202: システムは CloudWatch メトリクスによるエージェント実行監視を提供しなければならない 🟡 *note.md 運用注意事項から妥当な推測。Strands Agents のレイテンシー・エラー率の監視が推奨されている*

## Edge ケース

### エラー処理

- EDGE-SM-001: Strands Agent がマルチステップ推論中にタイムアウトした場合、処理中のステップを安全に中断し適切なエラーレスポンスを返却する 🟡 *note.md パフォーマンス注意事項から妥当な推測。エージェントの複合推論は Lambda タイムアウトを超過する可能性がある*
- EDGE-SM-002: ローカル開発環境で Ollama サーバーが起動していない場合、明確なエラーメッセージを返却し開発者に Ollama の起動を促す 🟡 *REQ-SM-101 のローカル開発要件から妥当な推測。Ollama 未起動は開発中に頻繁に発生するシナリオ*

### 境界値

- EDGE-SM-003: フィーチャーフラグ（`USE_STRANDS`）がアクティブなリクエスト処理中に切り替えられた場合、進行中のリクエストは開始時のフラグ値で最後まで処理を完了する 🟡 *REQ-SM-102/103 のフィーチャーフラグ要件から妥当な推測。リクエスト処理中の状態変更に対する防御的設計*
- EDGE-SM-004: エージェントツールが予期しない形式のレスポンスを返却した場合、ツール結果を破棄しエージェントはツールなしで推論を継続する 🟡 *REQ-SM-104 のグレースフルデグレード要件から妥当な推測。外部ツールの出力形式は保証されない*

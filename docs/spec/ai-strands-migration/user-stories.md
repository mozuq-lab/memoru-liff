## エピック1: AI基盤移行（Strands Agents SDK 導入）

### ストーリー 1.1: Strands Agents 経由でカードを生成する 🔵

**信頼性**: 🔵 *ユーザヒアリング・既存実装 `bedrock.py` より*

**私は** 学習者 **として**
**既存のカード生成機能が Strands Agents 経由で動作してほしい**
**そうすることで** AI基盤が最新のSDKで統一され、将来の機能拡張が容易になる

**関連要件**: REQ-SM-001, REQ-SM-002

**詳細シナリオ**:
1. カード生成画面でテキストを入力し「生成」をクリックする
2. Strands Agent が Bedrock プロバイダー経由で Claude を呼び出す
3. 生成されたカード（front/back/tags）が画面に表示される
4. カードの品質・レスポンス形式は現行と同等

**前提条件**:
- USE_STRANDS=true が設定されている
- Bedrock API へのアクセスが可能

**制約事項**:
- 既存の `GenerateCardsResponse` 形式を維持
- タイムアウトは30秒以内

**優先度**: Must Have

### ストーリー 1.2: フィーチャーフラグで新旧実装を切り替える 🔵

**信頼性**: 🔵 *ユーザヒアリング「段階的移行」選択より*

**私は** 運用者 **として**
**USE_STRANDS フラグで新旧AI実装を切り替えたい**
**そうすることで** 問題発生時に即座に旧実装にロールバックできる

**関連要件**: REQ-SM-102, REQ-SM-103, REQ-SM-201

**詳細シナリオ**:
1. Lambda 環境変数 `USE_STRANDS=false` → 既存の boto3 実装が使用される
2. Lambda 環境変数 `USE_STRANDS=true` → Strands Agents 実装が使用される
3. 切り替えはLambda 環境変数の変更で即時反映
4. どちらの実装でも API レスポンス形式は同一

**前提条件**:
- 新旧両方の実装コードがデプロイされている
- フィーチャーフラグ環境変数が設定されている

**優先度**: Must Have

### ストーリー 1.3: ローカル環境で Ollama を使ってAI機能をテストする 🔵

**信頼性**: 🔵 *ユーザヒアリング「Ollama統合（推奨）」選択より*

**私は** 開発者 **として**
**ローカル環境で Ollama を使ってAI機能をテストしたい**
**そうすることで** Bedrock API なしで完全なローカル開発ができる

**関連要件**: REQ-SM-101, REQ-SM-005

**詳細シナリオ**:
1. docker-compose.yaml に Ollama サービスが追加されている
2. `ENVIRONMENT=dev` の場合、Strands Agent が Ollama プロバイダーを使用する
3. カード生成・回答採点・学習アドバイスがローカルで動作する
4. ローカルモデルは軽量なもの（例: llama3.2, phi-3 等）を使用

**前提条件**:
- Docker / Docker Compose がインストールされている
- Ollama の Docker イメージがプルされている

**制約事項**:
- Ollama モデルの品質は Bedrock Claude に劣る可能性がある（開発確認用途）
- 初回起動時にモデルのダウンロードが必要

**優先度**: Must Have

## エピック2: 回答採点・AI評価機能

### ストーリー 2.1: 復習時にAIが回答を採点する 🔵

**信頼性**: 🔵 *ユーザヒアリング「回答採点・AI評価」選択より*

**私は** 学習者 **として**
**復習時に自分の回答をAIに評価してもらいたい**
**そうすることで** 自己評価のばらつきを減らし、より正確なSRSスケジューリングができる

**関連要件**: REQ-SM-003

**詳細シナリオ**:
1. 復習画面でカードの問題（front）が表示される
2. ユーザーが回答を入力する（テキスト入力）
3. 「AI評価」ボタンを押すと、AI が回答を採点する
4. 0-5 の SRS グレードと採点理由が表示される
5. ユーザーは AI 評価を受け入れるか、手動で上書きできる

**前提条件**:
- カードに front（問題）と back（解答）が設定されている
- AI サービスが利用可能

**制約事項**:
- レスポンスタイムは10秒以内を目標
- Lambda タイムアウトは60秒

**優先度**: Must Have

---

### ストーリー 2.2: AI採点結果に基づいて SRS スケジュールが更新される 🟡

**信頼性**: 🟡 *既存 SRS 実装（SM-2 アルゴリズム）と回答採点要件から妥当な推測*

**私は** 学習者 **として**
**AI 採点の結果が SRS スケジュールに自動反映されてほしい**
**そうすることで** 手動評価なしでも効率的な復習スケジュールが維持される

**関連要件**: REQ-SM-003

**詳細シナリオ**:
1. AI がユーザーの回答にグレード 3 を付与する
2. SM-2 アルゴリズムがグレード 3 で次回復習日を計算する
3. カードの `next_review_at` が更新される
4. 復習履歴（reviews テーブル）に AI 採点のフラグが記録される

**前提条件**:
- AI 採点が正常に完了している
- ユーザーが AI 採点結果を承認している

**優先度**: Should Have

**備考**: 既存の SM-2 アルゴリズム実装との統合方法は設計フェーズで詳細化

## エピック3: 学習アドバイス機能

### ストーリー 3.1: 学習履歴に基づいてAIアドバイスを受ける 🔵

**信頼性**: 🔵 *ユーザヒアリング「学習アドバイス」選択より*

**私は** 学習者 **として**
**学習履歴を分析してAIからアドバイスをもらいたい**
**そうすることで** 効率的な学習方法を知ることができる

**関連要件**: REQ-SM-004

**詳細シナリオ**:
1. ホーム画面または設定画面で「学習アドバイス」をタップする
2. AI が DynamoDB の復習履歴・カードデータを分析する
3. 弱点分野、おすすめ学習時間、復習ペースなどをアドバイスする
4. アドバイスは日本語（または英語）で表示される

**前提条件**:
- ユーザーに復習履歴データが存在する
- AI サービスが利用可能

**制約事項**:
- レスポンスタイムは15秒以内を目標
- Lambda タイムアウトは60秒

**優先度**: Must Have

---

### ストーリー 3.2: 弱点分野のカード追加提案を受ける 🟡

**信頼性**: 🟡 *学習アドバイス要件とツール統合要件から妥当な推測*

**私は** 学習者 **として**
**弱点分野に関する追加カードの提案を受けたい**
**そうすることで** 苦手な分野を重点的に学習できる

**関連要件**: REQ-SM-004, REQ-SM-006

**詳細シナリオ**:
1. 学習アドバイス画面で「弱点分野のカード提案」をタップする
2. AI が DynamoDB の復習履歴を参照（ツール呼び出し）する
3. 正答率の低いタグの関連カードを提案する
4. 提案されたカードを一括追加できる

**前提条件**:
- 十分な復習履歴データが存在する
- エージェントツール（DynamoDB参照）が利用可能

**優先度**: Could Have

**備考**: ツール統合（REQ-SM-006）の実装後に検討

## エピック4: エージェントツール統合

### ストーリー 4.1: Web検索で補足情報を含むカードを生成する 🟡

**信頼性**: 🟡 *ユーザヒアリング「外部ツールも含む」選択から具体的なユースケースは推測*

**私は** 学習者 **として**
**Web検索で最新情報を補完したカードを生成したい**
**そうすることで** より正確で最新の学習コンテンツが得られる

**関連要件**: REQ-SM-301, REQ-SM-006

**詳細シナリオ**:
1. カード生成時にオプションで「Web検索で補完」を有効化する
2. Strands Agent が Web 検索ツールで関連情報を取得する
3. 入力テキスト + Web 情報を組み合わせてカードを生成する
4. カードに情報ソースのリンクが付く

**前提条件**:
- Web 検索ツールが実装・登録されている
- インターネットアクセスが可能

**優先度**: Could Have

---

### ストーリー 4.2: URL からカードを生成する 🟡

**信頼性**: 🟡 *ユーザヒアリング「外部ツールも含む」選択から具体的なユースケースは推測*

**私は** 学習者 **として**
**URLを入力してWebページの内容からカードを生成したい**
**そうすることで** 記事やドキュメントの学習が簡単にできる

**関連要件**: REQ-SM-302, REQ-SM-006

**詳細シナリオ**:
1. カード生成画面で URL を入力する
2. Strands Agent が URL 読み込みツールでコンテンツを取得する
3. 取得したテキストからフラッシュカードを生成する
4. ソース URL がカードのメタデータに保存される

**前提条件**:
- URL 読み込みツールが実装・登録されている
- 対象 URL がアクセス可能

**優先度**: Could Have

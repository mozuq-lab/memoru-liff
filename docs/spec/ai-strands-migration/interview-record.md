# AI Strands Migration ヒアリング記録

**作成日**: 2026-02-23
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存のAI実装（boto3 直接呼び出しによる Bedrock カード生成）の設計文書・コードを分析し、Strands Agents SDK 移行に際しての不明点・方針を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: AI機能拡張のレベル

**質問日時**: 2026-02-23
**カテゴリ**: 追加要件
**背景**: 現在のAI機能はカード生成のみ（`backend/src/services/bedrock.py` の `BedrockService.generate_cards()`）。Strands Agents への移行に伴い、どの程度の機能拡張を行うか方針を確認する必要があった。

**質問**: "Strands Agentsへの移行で、どのレベルのAI機能拡張を想定していますか？"

**選択肢**:
- カード生成の高度化
- 複数AI機能の追加
- まずは基盤移行のみ

**回答**: **複数AI機能の追加** - カード生成に加え、回答採点、学習アドバイスなど新しいAI機能を追加

**信頼性への影響**:
- REQ-SM-003（回答採点・AI評価）を新規追加（🔵）
- REQ-SM-004（学習アドバイス）を新規追加（🔵）
- 移行の目的が「基盤移行」から「機能拡張を伴う移行」に明確化

---

### Q2: 移行中の後方互換性

**質問日時**: 2026-02-23
**カテゴリ**: アーキテクチャ
**背景**: boto3 直接呼び出しから Strands Agents への移行はリスクを伴う。既存の `GenerateCardsResponse` API契約（`backend/src/models/generate.py`）を破壊せず、安全に移行するための方針を確認する必要があった。

**質問**: "移行中の後方互換性について、どの方針を希望しますか？"

**選択肢**:
- 段階的移行（推奨）
- 一括置き換え
- 並行運用

**回答**: **段階的移行（推奨）** - 環境変数フラグで新旧実装を切り替え可能にし、段階的にロールアウト

**信頼性への影響**:
- REQ-SM-102（USE_STRANDS=true で Strands 使用）を新規追加（🔵）
- REQ-SM-103（USE_STRANDS=false で boto3 フォールバック）を新規追加（🔵）
- REQ-SM-201（新旧共存フェーズ）を新規追加（🔵）
- 移行戦略が「フィーチャーフラグによる段階的移行」に確定

---

### Q3: 追加したいAI機能の詳細

**質問日時**: 2026-02-23
**カテゴリ**: 未定義部分詳細化
**背景**: Q1 で「複数AI機能の追加」が回答されたため、具体的にどの機能を優先するか確認する必要があった。設計文書（`docs/design/memoru-liff/architecture.md`）では「将来の回答採点機能」が言及されていた。

**質問**: "追加したいAI機能について、具体的にどの機能を優先したいですか？（複数選択可）"

**選択肢（複数選択）**:
- 回答採点・AI評価
- 学習アドバイス
- カード改善提案
- テキスト要約・分析

**回答**: **回答採点・AI評価** と **学習アドバイス** を選択

**信頼性への影響**:
- REQ-SM-003（回答採点）の信頼性が 🟡 → 🔵 に向上（具体的な機能選択として確定）
- REQ-SM-004（学習アドバイス）の信頼性が 🟡 → 🔵 に向上
- カード改善提案・テキスト要約はスコープ外と確定

### Q4: ツール機能の活用

**質問日時**: 2026-02-23
**カテゴリ**: 技術選択
**背景**: Strands Agents SDK の主要な特徴の一つが `@tool` デコレータによるツール統合。エージェントが DynamoDB データ参照や Web 検索などの外部ツールを使用する範囲を確認する必要があった。

**質問**: "Strands Agentsのツール機能（Web検索、ファイル読み込み等）の活用は検討していますか？"

**選択肢**:
- 当面は不要
- 基本的なツールのみ
- 外部ツールも含む

**回答**: **外部ツールも含む** - Web検索、画像解析、URL読み込みなどの外部ツールも統合

**信頼性への影響**:
- REQ-SM-006（ツール統合フレームワーク）を新規追加（🔵）
- REQ-SM-301（Web検索ツール）を新規追加（🔵）
- REQ-SM-302（URL読み込みツール）を新規追加（🔵）
- REQ-SM-303（画像解析ツール）を新規追加（🔵）

---

### Q5: ローカル開発環境でのAIテスト

**質問日時**: 2026-02-23
**カテゴリ**: 技術制約
**背景**: 現在のローカル環境では Amazon Bedrock にアクセスできず、`POST /cards/generate` がエラーになる（TASK-0051 E2E確認で確認済み）。Strands Agents は Ollama プロバイダーをサポートしており、ローカルでの AI 動作を実現できる可能性がある。

**質問**: "ローカル開発環境でのAI機能テストについて、どの方針を希望しますか？"

**選択肢**:
- Ollama統合（推奨）
- Bedrockのみ
- モック対応

**回答**: **Ollama統合（推奨）** - Strands Agents の Ollama プロバイダーを使用し、ローカルで完全な AI 動作を実現

**信頼性への影響**:
- REQ-SM-101（dev環境で Ollama 使用）を新規追加（🔵）
- REQ-SM-005（マルチモデルサポート）の必要性が確定（🔵）
- ローカル開発体験の大幅な向上が期待される

---

### Q6: Lambda タイムアウト設定

**質問日時**: 2026-02-23
**カテゴリ**: パフォーマンス
**背景**: Strands Agents はマルチステップ推論を行うため、現在の Lambda グローバルタイムアウト 30秒（`template.yaml` Globals.Function.Timeout: 30）では不足する可能性がある。特に回答採点や学習アドバイスは DynamoDB からの履歴読み込み + AI推論 を行うため、より長い処理時間が必要。

**質問**: "Lambda関数のタイムアウトについて、現在の30秒から変更は必要ですか？"

**選択肢**:
- 30秒のまま
- 60秒に拡張
- 機能別に設定

**回答**: **機能別に設定** - カード生成は30秒、回答採点は60秒など機能別にタイムアウトを設定

**信頼性への影響**:
- REQ-SM-401（機能別タイムアウト）を新規追加（🔵）
- Lambda 関数分離の設計方針が確定（機能別に異なるタイムアウトを持つ関数構成）

## ヒアリング結果サマリー

### 確認できた事項

- Strands Agents 移行の目的は「複数AI機能の追加」を伴う拡張的移行
- 優先する新AI機能: 回答採点・AI評価、学習アドバイス
- 移行戦略: フィーチャーフラグ（USE_STRANDS）による段階的移行
- 外部ツール（Web検索、画像解析、URL読み込み）も統合する方針
- ローカル開発では Ollama プロバイダーで Bedrock 不要の完全ローカル動作
- Lambda タイムアウトは機能別に設定（カード生成30秒、採点/アドバイス60秒）

### 追加/変更要件

- REQ-SM-003: 回答採点・AI評価機能（新規 🔵）
- REQ-SM-004: 学習アドバイス機能（新規 🔵）
- REQ-SM-005: マルチモデルサポート（新規 🔵）
- REQ-SM-006: ツール統合フレームワーク（新規 🔵）
- REQ-SM-101: dev環境 Ollama プロバイダー使用（新規 🔵）
- REQ-SM-102/103: フィーチャーフラグによる段階的移行（新規 🔵）
- REQ-SM-201: 新旧実装共存フェーズ（新規 🔵）
- REQ-SM-301〜303: 外部ツール統合（新規 🔵）
- REQ-SM-401: 機能別 Lambda タイムアウト設定（新規 🔵）

### 残課題

- Strands Agents SDK の具体的なバージョン選定（最新安定版の確認が必要）
- Ollama 対応モデルの選定（ローカル開発用、軽量かつ日本語対応）
- 外部ツールの具体的な実装詳細（後続の設計フェーズで決定）
- 回答採点のグレーディング基準の詳細設計（SRS グレード 0-5 への変換ロジック）
- 学習アドバイスの分析対象データ範囲と分析ロジックの詳細設計

# code-quality-improvements ヒアリング記録

**作成日**: 2026-03-01
**ヒアリング実施**: コードレビュー指摘事項に基づく差分ヒアリング

## ヒアリング目的

Claude Opus 4.6 + OpenAI Codex によるコードレビュー結果（25件）に基づき、
改善要件のスコープ・仕様判断・対応方針を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: 改善要件のスコープ

**カテゴリ**: 優先順位
**背景**: 25件の指摘を全て対応するか、重大度でフィルタするかの判断が必要

**回答**: Critical + High + Medium の 17件を対応する

**信頼性への影響**:

- 全 17件の要件が対応スコープとして確定（🔵）
- Low 6件・Info 2件は対象外

---

### Q2: Undo 時の reviews_table の扱い

**カテゴリ**: 未定義部分詳細化
**背景**: H-08 で Codex が「reviews_table を巻き戻すべき」と指摘。現在の実装は cards_table のみ復元しており、reviews_table のレコードは残存する。これが仕様なのかバグなのか不明だった。

**回答**: 仕様として明文化する。「Undo は SRS パラメータのみ復元」と明文化。

**信頼性への影響**:

- H-08 の要件内容が「reviews_table 巻き戻し」→「仕様明文化（コメント追加）」に変更
- 信頼性が 🟡 → 🔵 に向上

---

### Q3: JWT dev フォールバックの対応方針

**カテゴリ**: 既存設計確認
**背景**: CR-01 でレビュー結果から複数の対応レベルが提案されていた。最低限のログ追加か、環境変数の強化まで行うかの判断が必要。

**回答**: ログ追加 + 二重実装統合 + 環境変数強化（AWS_SAM_LOCAL 等との組み合わせ）

**信頼性への影響**:

- CR-01 の要件が具体化し、信頼性が 🔵 に確定
- 実装方針が明確化

---

### Q4: _get_next_due_date の動作確認

**カテゴリ**: 既存設計確認
**背景**: M-06 で Claude が「next_review_at > now の条件がない」と指摘。ただし get_due_cards で due_cards が空だった場合にのみ呼ばれるため、期限切れカードは存在しない前提。現状で問題ないか確認が必要。

**回答**: 条件追加すべき。next_review_at > now の条件を追加して明示的に将来日のみ取得する。

**信頼性への影響**:

- M-06 の要件が「確認推奨」→「条件追加（実装変更）」に確定
- 信頼性が 🟡 → 🔵 に向上

---

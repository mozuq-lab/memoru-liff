# interval-adjust（復習間隔調整） 要件定義書

## 概要

カード詳細画面から復習間隔（interval）を手動で調整できる機能を実装する。SM-2アルゴリズムによる自動計算に加え、ユーザーが「このカードはもう少し頻繁に復習したい」「もっと間隔を空けたい」といった要望に対応する。interval変更時に next_review_at を自動再計算し、ease_factor と repetitions は変更しない。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: 設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: 設計文書・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: 設計文書・ユーザヒアリングにない推測による要件

### 通常要件

- REQ-001: システムは、カード詳細画面でプリセットボタン（1日, 3日, 7日, 14日, 30日）を表示しなければならない 🔵 *ユーザヒアリング UI形式・プリセット値より*
- REQ-002: システムは、ユーザーがプリセットボタンを押下した時に `PUT /cards/:card_id` API でintervalを更新しなければならない 🔵 *ユーザヒアリング API設計より*
- REQ-003: システムは、interval変更時に next_review_at を「現在日時 + interval日」で自動再計算しなければならない 🔵 *前回のSM-2説明での合意（Aの案）より*
- REQ-004: システムは、interval変更時に ease_factor と repetitions を変更してはならない 🔵 *ユーザヒアリング repetitions確認より*

### 条件付き要件

- REQ-101: interval値が1未満の場合、システムはバリデーションエラーを返さなければならない 🔵 *ユーザヒアリング 値の制約より*
- REQ-102: interval値が365を超える場合、システムはバリデーションエラーを返さなければならない 🔵 *ユーザヒアリング 値の制約より*
- REQ-103: 間隔調整のAPI呼び出しが失敗した場合、システムはエラーメッセージを表示し、UIの状態を変更前に戻さなければならない 🟡 *既存実装のエラーハンドリングパターンから妥当な推測*

### 状態要件

- REQ-201: カード詳細画面の表示モードにある場合、システムはプリセットボタンを表示しなければならない 🔵 *ユーザヒアリング UI配置より*
- REQ-202: 間隔調整のAPI呼び出し中にある場合、システムはプリセットボタンを無効化しなければならない 🟡 *既存UI実装パターン（isSubmitting）から妥当な推測*
- REQ-203: 間隔調整が成功した場合、システムはカード詳細画面のメタ情報（復習間隔・次回復習日）を即座に更新しなければならない 🔵 *CardDetailPageの既存更新パターンより*

### 制約要件

- REQ-401: システムは、既存の `PUT /cards/:card_id` エンドポイントに interval フィールドを追加することで間隔調整を実現しなければならない 🔵 *ユーザヒアリング API設計より*
- REQ-402: システムは、interval フィールドが指定された場合のみ next_review_at を再計算し、front/back等の更新時には SRS パラメータを変更してはならない 🟡 *既存 update_card の動作パターンから妥当な推測*
- REQ-403: システムは、interval 変更を review_history に記録してはならない（復習ではないため） 🟡 *SM-2アルゴリズムの設計意図から妥当な推測*

## 非機能要件

### パフォーマンス

- NFR-001: interval更新APIのレスポンスタイムは500ms以内であること 🟡 *既存API性能要件から妥当な推測*

### ユーザビリティ

- NFR-201: プリセットボタンは現在のinterval値をハイライト表示すること 🟡 *ユーザビリティ上の妥当な推測*
- NFR-202: プリセットボタンのタップ領域は44px以上を確保すること 🔵 *既存UIの共通パターンより*
- NFR-203: 調整成功時にフィードバック（成功メッセージ）を表示すること 🔵 *CardDetailPageの既存successMessageパターンより*

### アクセシビリティ

- NFR-301: プリセットボタンに適切なaria-labelを設定すること 🔵 *既存UIのアクセシビリティパターンより*

## Edgeケース

### エラー処理

- EDGE-001: ネットワークエラー時にエラーメッセージを表示し、元の値を復元する 🟡 *既存エラーハンドリングパターンから妥当な推測*
- EDGE-002: 同一カードに対して連続で間隔調整した場合、最新の値が反映される 🟡 *DynamoDB更新の性質から妥当な推測*

### 境界値

- EDGE-101: interval=1（最小値）で正常に更新できること 🔵 *ユーザヒアリング 値の制約より*
- EDGE-102: interval=365（最大値）で正常に更新できること 🔵 *ユーザヒアリング 値の制約より*
- EDGE-103: 未復習カード（repetitions=0, interval=0）に対して間隔調整ができること 🟡 *初期状態カードへの操作として妥当な推測*
